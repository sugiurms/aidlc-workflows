# INCEPTION フェーズ — ステージ詳細

**目的**: 計画、要件収集、アーキテクチャの決定

**フォーカス**: 何を構築するか、なぜ構築するかを決定する

**対象ステージ**:

- ワークスペース検出（常時）
- リバースエンジニアリング（条件付き — ブラウンフィールドのみ）
- 要件分析（常時 — アダプティブな詳細度）
- ユーザーストーリー（条件付き）
- ワークフロー計画（常時）
- アプリケーション設計（条件付き）
- ユニット生成（条件付き）

各ステージは SKILL.md の「標準ステージ実行パターン」に従って実行する。以下はそれぞれのステージ固有の参照ファイル、実行内容、承認待機要件。

---

## ワークスペース検出（常時実行）

**参照**: `aws-aidlc-rule-details/inception/workspace-detection.md`

**実行内容**:

- 既存の aidlc-state.md を確認する（見つかった場合は再開）
- 既存のコードのワークスペースをスキャンする
- ブラウンフィールドかグリーンフィールドかを決定する
- 既存のリバースエンジニアリングアーティファクトを確認する

**次フェーズの決定**: ブラウンフィールドでアーティファクトがない場合はリバースエンジニアリングへ、それ以外は要件分析へ。

**承認待機**: 不要。完了メッセージを提示した後、自動的に次のフェーズに進む（ここで承認を求めると再開のたびに冗長になるため）。

---

## リバースエンジニアリング（条件付き — ブラウンフィールドのみ）

**実行条件**:

- 既存のコードベースが検出された、かつ
- 以前のリバースエンジニアリングアーティファクトが見つからない

**スキップ条件**:

- グリーンフィールドプロジェクト、または
- 以前のリバースエンジニアリングアーティファクトが存在する

**参照**: `aws-aidlc-rule-details/inception/reverse-engineering.md`

**実行内容**:

- すべてのパッケージとコンポーネントを分析する
- ビジネストランザクションをカバーするシステム全体のビジネス概要を生成する
- アーキテクチャドキュメントを生成する
- コード構造ドキュメントを生成する
- API ドキュメントを生成する
- コンポーネントインベントリを生成する
- ビジネストランザクションがコンポーネント全体でどのように実装されているかを示すインタラクション図を生成する
- 技術スタックドキュメントを生成する
- 依存関係ドキュメントを生成する

**承認待機**: 必要。詳細な完了メッセージを提示し、ユーザーが確認するまで次ステージへ進まない（メッセージフォーマットは参照先ファイルを参照）。リバースエンジニアリングの結果は後続の全ステージの土台になるため、ここで認識齟齬があると下流全部が歪む。

---

## 要件分析（常時実行 — アダプティブな詳細度）

常に実行されるが、リクエストの明確さと複雑さに基づいて詳細度が変わる:

- **最小**: シンプルで明確なリクエスト — 意図分析を文書化するだけ
- **標準**: 通常の複雑さ — 機能要件と非機能要件を収集する
- **包括的**: 複雑で高リスク — トレーサビリティを含む詳細な要件

**参照**: `aws-aidlc-rule-details/inception/requirements-analysis.md`

**実行内容**:

- リバースエンジニアリングアーティファクトを読み込む（ブラウンフィールドの場合）
- ユーザーリクエストを分析する（意図分析）
- 必要な要件の詳細度を決定する
- 現在の要件を評価する
- 確認質問をする（必要な場合）
- 要件ドキュメントを生成する

**承認待機**: 必要。参照先ファイルの詳細な承認フォーマットに従う。要件が曖昧なまま下流に進むと手戻りコストが大きいため。

---

## ユーザーストーリー（条件付き）

多要素分析を使用してユーザーストーリーが価値を追加するかどうかを決定する。

**常に実行**（高優先度の指標）:

- 新しいユーザー向け機能
- ユーザーワークフローまたはインタラクションに影響する変更
- 複数のユーザータイプまたはペルソナが関与している
- 受け入れ基準のニーズを持つ複雑なビジネス要件
- クロスファンクショナルなチームコラボレーションが必要
- 顧客向け API またはサービスの変更
- 新しい製品機能または機能強化

**おそらく実行**（中優先度 — 複雑さを評価）:

- 既存のユーザー向け機能への変更
- ユーザーエクスペリエンスに間接的に影響するバックエンドの変更
- ユーザーワークフローに影響する統合作業
- ユーザーに見えるメリットがあるパフォーマンスの改善
- ユーザーインタラクションに影響するセキュリティ強化
- ユーザーデータまたはレポートに影響するデータモデルの変更

中優先度のケースでは、以下のいずれかに該当する場合に実行する:

- リクエストが複数のコンポーネントまたはサービスを含む
- 変更が複数のユーザータッチポイントにまたがる
- ビジネスロジックが複雑か複数のシナリオを持つ
- ストーリーが明確にできる曖昧さが要件にある
- 変更が複数のユーザージャーニーに影響する
- 変更に重大なビジネスインパクトまたはリスクがある

**スキップ条件**（低優先度 — シンプルなケース）:

- ユーザーへの影響がゼロの純粋な内部リファクタリング
- 明確で孤立したスコープのシンプルなバグ修正
- ユーザー向けの影響がないインフラストラクチャの変更
- 機能変更のない技術的負債のクリーンアップ
- 開発者ツールまたはビルドプロセスの改善
- ドキュメントのみの更新

**判断の指針**: ビジネスステークホルダーが関与する、ユーザー受け入れテストが必要、複数の実装アプローチがある、チームの共通理解から恩恵を受ける、要件の明確さが価値あるプロジェクト — こうしたケースでは迷ったら含める。

**評価プロセス**:

1. リクエストの複雑さとスコープを分析する
2. ユーザーへの影響（直接または間接）を特定する
3. ビジネスコンテキストとステークホルダーのニーズを評価する
4. チームコラボレーションのメリットを考慮する
5. ボーダーラインのケースでは含有にデフォルトする

**参照**: `aws-aidlc-rule-details/inception/user-stories.md`

**実行内容**（2 パート構成）:

- パート 1 — 計画: 質問を含むストーリー計画を作成し、ユーザーの回答を待ち、曖昧さを分析して承認を得る
- パート 2 — 生成: 承認された計画を実行してストーリーとペルソナを生成する

要件分析が実行された場合、ストーリーはそれらの要件を参照して基に構築する。ブラウンフィールドの場合はリバースエンジニアリングアーティファクトも読み込む。適切な詳細度（最小/標準/包括的）で実行する。

**承認待機**: 各パートで必要。参照先ファイルの詳細なステップの承認フォーマットに従う。

## ワークフロー計画（常時実行）

**参照**: `aws-aidlc-rule-details/inception/workflow-planning.md`、および `aws-aidlc-rule-details/common/content-validation.md`

**事前コンテキストの読み込み**:

- リバースエンジニアリングアーティファクト（ブラウンフィールドの場合）
- 意図分析
- 要件（実行された場合）
- ユーザーストーリー（実行された場合）

**実行内容**:

- 実行するフェーズを決定する
- 各フェーズの詳細度レベルを決定する
- マルチパッケージ変更シーケンスを作成する（ブラウンフィールドの場合）
- ワークフロー可視化を生成する（書き込む前に Mermaid 構文を検証する）

ファイル作成の前に content-validation.md のルールに従ってコンテンツを検証する。構文不正な Mermaid やエスケープ漏れは下流の読み取りエラーにつながるため。

**承認待機**: 必要。workflow-planning.md のステップ 9 の言語を使用して推奨事項を提示し、ユーザーが推奨をオーバーライドできるコントロールを強調する。

---

## アプリケーション設計（条件付き）

**実行条件**:

- 新しいコンポーネントまたはサービスが必要
- コンポーネントメソッドとビジネスルールの定義が必要
- サービス層設計が必要
- コンポーネントの依存関係の明確化が必要

**スキップ条件**:

- 既存のコンポーネント境界内での変更
- 新しいコンポーネントやメソッドなし
- 純粋な実装の変更

**参照**: `aws-aidlc-rule-details/inception/application-design.md`

**実行内容**:

- ブラウンフィールドの場合、リバースエンジニアリングアーティファクトを読み込む
- 適切な詳細度（最小/標準/包括的）で設計を実施する

**承認待機**: 必要。詳細な完了メッセージのフォーマットは参照先ファイルを参照。

---

## ユニット生成（条件付き）

**実行条件**:

- システムを複数の作業単位に分解する必要がある
- 複数のサービスまたはモジュールが必要
- 構造化された分解が必要な複雑なシステム

**スキップ条件**:

- 単一のシンプルなユニット
- 分解が不要
- 直接的な単一コンポーネントの実装

**参照**: `aws-aidlc-rule-details/inception/units-generation.md`

**実行内容**:

- ブラウンフィールドの場合、リバースエンジニアリングアーティファクトを読み込む
- 適切な詳細度（最小/標準/包括的）でユニット分解を実施する

**承認待機**: 必要。詳細な完了メッセージのフォーマットは参照先ファイルを参照。
