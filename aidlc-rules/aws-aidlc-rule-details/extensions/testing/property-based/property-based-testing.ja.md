# プロパティベーステストルール

## 概要

これらのプロパティベーステスト（PBT）ルールは、適用される AI-DLC フェーズにまたがって適用される横断的制約です。識別可能なプロパティを持つコードがプロパティベースの技法を使用してテストされることを保証し、従来の例ベーステストを（置き換えるのではなく）補完します。

プロパティベーステストは、すべての有効な入力に対して成立しなければならない不変条件を定義し、フレームワークを使用してランダムな入力を生成して反例を探します。失敗が見つかると、フレームワークは入力を最小限の再現ケースに縮小します。このアプローチは、例ベーステストが見逃しがちなエッジケースや微妙なバグを発見します。

**強制適用**: 各該当ステージにおいて、モデルはステージ完了メッセージをユーザーに提示する前に、これらのルールへの準拠を検証しなければなりません。

### PBT 発見のブロッキング動作

**ブロッキング PBT 発見**とは:
1. 発見内容は PBT ルール ID と説明を含む「PBT Findings」セクションとしてステージ完了メッセージに記載しなければならない
2. すべてのブロッキング発見が解消されるまで、ステージは「次のステージへ進む」オプションを提示してはならない
3. モデルは「変更を要求」オプションのみを提示し、変更が必要な内容を明確に説明しなければならない
4. 発見内容は PBT ルール ID、説明、ステージのコンテキストとともに `aidlc-docs/audit.md` に記録しなければならない

PBT ルールが現在のプロジェクトまたはユニットに適用されない場合（例: ステートフルコンポーネントが存在しない場合の PBT-06）は、準拠サマリーで **N/A** とマークしてください — これはブロッキング発見ではありません。

### デフォルトの強制適用

このドキュメントのすべてのルールはデフォルトで**ブロッキング**です。ルールの検証基準が満たされない場合、それはブロッキング PBT 発見です — 上記で定義されたブロッキング発見の動作に従ってください。

### 部分強制適用モード

オプトイン時にユーザーが**部分的**な強制適用を選択した場合、PBT-02、PBT-03、PBT-07、PBT-08、PBT-09 のルールのみが強制適用されます。その他のすべてのルールはアドバイザリー（非ブロッキング）として扱われます。強制適用モードを `aidlc-docs/aidlc-state.md` の `## Extension Configuration` 下に記録してください。

### 検証基準のフォーマット

このドキュメントの検証項目は、準拠チェックを説明するプレーンな箇条書きです。各項目はレビュー時に準拠または非準拠として評価してください。

## ルール PBT-01: 設計フェーズでのプロパティ特定

**ルール**: ビジネスロジック、データ変換、またはアルゴリズム操作を含むすべてのユニットは、機能設計ステージでテスト可能なプロパティを分析しなければなりません。分析では以下のプロパティカテゴリのどれが適用されるかを特定しなければなりません:

| カテゴリ | 説明 | 例 |
|---|---|---|
| ラウンドトリップ | 逆演算と組み合わせた操作が元の値を返す | シリアライズ → デシリアライズ = 同一 |
| 不変条件 | 変換が測定可能な特性を保持する | ソートはコレクションのサイズと要素を保持する |
| 冪等性 | 操作を 2 回適用すると 1 回と同じ結果になる | dedup(dedup(list)) = dedup(list) |
| 可換性 | 異なる操作順序が同じ結果を生成する | add(a, b) = add(b, a) |
| オラクル | 参照実装または簡略化モデルで結果を検証できる | 最適化アルゴリズム対ブルートフォース |
| 帰納法 | より小さな入力で証明されたプロパティがより大きな入力に適用される | 再帰的構造、分割統治 |
| 検証容易性 | 結果の計算は難しいが確認は容易 | 迷路ソルバーの出力を辿って検証できる |

特定されたプロパティは、ユニットの機能設計アーティファクトに文書化し、PBT テスト要件としてコード生成に引き継がなければなりません。

**検証**:
- 機能設計アーティファクトに、コンポーネントごとに特定されたプロパティをリストした「Testable Properties」セクションが含まれている
- 各特定されたプロパティが上記のカテゴリの 1 つを参照している
- 識別可能なプロパティがないコンポーネントには、「No PBT properties identified」と簡潔な理由が明示されている
- コード生成計画中にプロパティリストが参照される

## ルール PBT-02: ラウンドトリッププロパティ

**ルール**: 論理的な逆演算を持つ操作は、ラウンドトリップを検証するプロパティベーステストを持たなければなりません。これには以下が含まれますが、これに限定されません:
- シリアライゼーション/デシリアライゼーション（JSON、XML、Protobuf、バイナリフォーマット）
- エンコード/デコード（Base64、URL エンコーディング、圧縮）
- パース/フォーマット（日付のパース、数値のフォーマット、構造化入力によるテンプレートレンダリング）
- 暗号化/復号（キーが利用可能な場合）
- データベースへの書き込み/読み取り（I/O 自体ではなく、データ変換レイヤー用）
- すべての有効な `x` に対して `f_inverse(f(x)) = x` である関数ペア

プロパティベーステストは、ドメインに適したジェネレーター（PBT-07 参照）を使用してランダムな有効入力を生成し、ラウンドトリップが元の入力と等しい値を生成することを表明しなければなりません。

**検証**:
- すべてのシリアライゼーション/デシリアライゼーションペアにラウンドトリッププロパティテストがある
- すべてのエンコード/デコードペアにラウンドトリッププロパティテストがある
- すべてのパース/フォーマットペアにラウンドトリッププロパティテストがあるか、変換が不可逆的である理由を文書化している
- ラウンドトリップテストがハードコードされた例ではなく生成された入力を使用する
- 不可逆変換（例: 精度が失われる浮動小数点フォーマット）は許容偏差を文書化し、許容範囲内でテストする

## ルール PBT-03: 不変条件プロパティ

**ルール**: 文書化された不変条件を持つ関数は、生成された入力にまたがってそれらの不変条件が成立することを検証するプロパティベーステストを持たなければなりません。一般的な不変条件には以下が含まれます:
- **サイズ保持**: 出力コレクションの入力と同じサイズ（例: map、sort）
- **要素保持**: 出力に入力とまったく同じ要素が（順序は変わるかもしれないが）含まれる（例: sort、shuffle）
- **順序保証**: 出力が順序制約を満たす（例: sort が非減少順を生成する）
- **範囲制約**: 出力値が定義された範囲に収まる（例: normalize が [0, 1] の値を生成する）
- **型保持**: すべての有効な入力に対して出力型が期待される型と一致する
- **ビジネスルールの不変条件**: 常に成立しなければならないドメイン固有のルール（例: 「有効なトランザクション後にアカウント残高がマイナスになることはない」「割引がアイテム価格を超えることはない」）

**検証**:
- 各文書化された不変条件に対応するプロパティベーステストがある
- 不変条件テストが境界値を含む広い範囲の入力を生成する
- 機能設計で特定されたビジネスルールの不変条件が PBT でカバーされている
- 不変条件テストが例ベーステストと同じアサーションを重複させない — 特定のケースではなく一般的なルールをテストする

---

## ルール PBT-04: 冪等性プロパティ

**ルール**: 冪等性を主張または要求する操作は、それを証明するプロパティベーステストを持たなければなりません。テストはすべての有効な生成入力に対して `f(f(x)) = f(x)` を検証しなければなりません。これは以下に適用されます:
- 冪等として文書化された API エンドポイント（PUT、DELETE）
- データ正規化またはサニタイゼーション関数
- キャッシュ投入操作
- 重複排除ロジック
- 設定の適用（設定を 2 回適用しても状態が変わらないこと）
- 少なくとも 1 回の配信システムでのメッセージ処理

**検証**:
- 冪等として文書化されたすべての操作に `f(f(x)) = f(x)` を表明する PBT がある
- 冪等性テストがドメインに適したジェネレーターを使用する（プリミティブのみではない）
- ステートフルな操作の場合、テストが単一適用と繰り返し適用後の観測可能な状態の等価性を検証する

---

## ルール PBT-05: オラクルおよびモデルベーステスト

**ルール**: 参照実装、簡略化モデル、または既知の正しいアルゴリズムが存在する場合、プロパティベーステストはテスト対象システムとオラクルを比較しなければなりません。これは以下に適用されます:
- 既知のブルートフォースバージョンを置き換える最適化アルゴリズム
- レガシー実装を置き換えるリファクタリングされたコード
- 逐次バージョンと比較される並列/並行実装
- よく知られたアルゴリズムのカスタム実装（ソート、検索、グラフ走査）
- 参照データベースと比較される新しいクエリエンジン

プロパティベーステストはランダムな有効入力を生成し、すべての生成入力に対してテスト対象システムがオラクルと等価な結果を生成することを表明しなければなりません。

**検証**:
- 参照実装が存在する（または簡単に作成できる）場合、オラクル PBT が存在する
- オラクルテストが通常、境界、および敵対的なケースをカバーする多様な入力を生成する
- 等価性が正確に定義されている（完全等値、構造的等値、または文書化された許容範囲）
- オラクルが存在しない場合、このルールは理由とともに N/A とマークされる

---

## ルール PBT-06: ステートフルプロパティテスト

**ルール**: 変更可能な状態を管理するコンポーネントは、ステートフルプロパティテストの評価を受けなければなりません。ステートフル PBT はシステムに対するランダムなコマンド（操作）のシーケンスを生成し、各ステップ後に不変条件が成立することを検証します。これは以下に適用されます:
- インメモリキャッシュとデータストア
- ステートマシンとワークフローエンジン
- キューとバッファの実装
- セッション管理システム
- ショッピングカート、注文パイプライン、および同様のステートフルなビジネスオブジェクト
- 操作の結果が以前の操作に依存するコンポーネント

ステートフル PBT では以下が必要です:
- テスト対象システムを反映した簡略化モデル（参照状態）を定義する
- 有効なコマンドのランダムシーケンスを生成する（追加、削除、更新、クエリなど）
- 各コマンドを実際のシステムとモデルの両方に対して実行する
- 各コマンド後に、システムとモデル間で観測可能な状態またはクエリ結果が一致することを表明する
- 空のシーケンスを含む様々な長さのシーケンスをテストする

**検証**:
- 機能設計で特定されたステートフルコンポーネントがステートフル PBT を持つか、適用できない理由を文書化している
- 比較のための簡略化モデルが定義されている
- コマンドジェネレーターが現実的なパラメーター分布を持つ有効な操作シーケンスを生成する
- 不変条件がシーケンスの最後だけでなく、各コマンド後にチェックされる
- ステートフルコンポーネントが存在しない場合、このルールは N/A とマークされる

---

## ルール PBT-07: ジェネレーターの品質

**ルール**: プロパティベーステストは、現実的な構造化入力を生成するドメイン固有のジェネレーターを使用しなければなりません — プリミティブ型だけを使用してはなりません。不十分なジェネレーター（例: メールフィールドへのランダム文字列、年齢フィールドへの制限されていない整数）は意味のないテストケースを生成し、実際のバグを見逃します。

ジェネレーターの要件:
- **ドメイン型**: ビジネス制約（有効なメールフォーマット、正の金額、有効な日付範囲）を考慮したドメインオブジェクト（例: User、Order、Transaction）のカスタムジェネレーターを作成しなければならない
- **制約されたプリミティブ**: ドメインが要求する場合、数値ジェネレーターは現実的な範囲に制約しなければならない
- **構造化データ**: 複雑な入力（ネストされたオブジェクト、ドメインオブジェクトのリスト）のジェネレーターは構造的に有効なデータを生成しなければならない
- **エッジケースの包含**: ジェネレーターは通常値とともに境界値（空のコレクション、ゼロ、最大値、Unicode 文字列）を含めるように設定すべきである
- **再利用性**: ドメインジェネレーターはテストファイル全体で重複させず、再利用可能なテストユーティリティとして定義すべきである

**検証**:
- ドメイン型パラメーターに対してプリミティブジェネレーターのみ（例: `st.integers()` のみ）を使用する PBT がない
- PBT で使用されるドメインオブジェクトのカスタムジェネレーターが存在する
- ジェネレーターが文書化されたビジネス制約（例: 正の金額、有効なフォーマット）を考慮する
- 複数のテストが同じドメイン型を共有する場合、ジェネレーター定義が集中化され再利用可能である

---

## ルール PBT-08: 縮小と再現性

**ルール**: すべてのプロパティベーステストは縮小と決定論的な再現性をサポートしなければなりません。

- **縮小**: プロパティが失敗した場合、PBT フレームワークは失敗した入力を最小限の再現ケースに自動的に縮小しなければならない。技術的な理由（例: 縮小が外部サービス呼び出しを含む統合テストと互換性がない）が文書化されていない限り、テストはフレームワークの縮小メカニズムを無効化またはバイパスしてはならない。
- **再現性**: すべての PBT 実行はシード値によって再現可能でなければならない。正確な失敗シナリオを再生できるように、失敗時にシードをログに記録しなければならない。CI 設定は、決定論的な実行のために固定シードを使用するか、失敗後の再現のためにすべての実行でランダムシードをログに記録しなければならない。
- **CI 統合**: PBT はプロジェクトの CI パイプラインに含めなければならない。不安定な PBT 失敗（コードの変更なしに再試行で通過するテスト）は、抑制するのではなく調査しなければならない。

**検証**:
- PBT フレームワークの縮小が有効（オーバーライドや無効化されていない）
- 失敗時のテスト出力にシード値と縮小された最小限の失敗入力が含まれている
- CI 設定がすべての PBT 実行でシードをログに記録するか、固定シードを使用する
- 文書化された正当な理由なく CI から除外される PBT がない
- 不安定な PBT 失敗が追跡・調査されており、黙って再試行されない

---

## ルール PBT-09: フレームワーク選択

**ルール**: プロジェクトは主要な言語に適したプロパティベーステストフレームワークを選択して設定しなければなりません。フレームワークは以下をサポートしなければなりません:
- ドメイン型のカスタムジェネレーター/ストラテジー
- 失敗ケースの自動縮小
- シードベースの再現性
- プロジェクトの既存のテストランナーとの統合

言語別の推奨フレームワーク（非網羅的）:

| 言語 | フレームワーク | 備考 |
|---|---|---|
| Python | Hypothesis | 成熟、優れた縮小、Django 統合 |
| JavaScript / TypeScript | fast-check | Jest、Vitest、Mocha と統合 |
| Java | jqwik | JUnit 5 統合、ステートフルテストサポート |
| Kotlin | Kotest Property Testing | Kotest フレームワーク統合 |
| Scala | ScalaCheck | SBT 統合、広く採用されている |
| Rust | proptest | マクロベース、優れた縮小 |
| Go | rapid | 軽量、慣用的な Go |
| Haskell | QuickCheck | 最初の PBT フレームワーク |
| C# / .NET | FsCheck | xUnit、NUnit と動作 |
| Erlang / Elixir | PropEr / StreamData | OTP 対応、ステートフルテスト |

選択されたフレームワークは技術スタック決定に文書化され、プロジェクトの依存関係に含まれなければなりません。

**検証**:
- PBT フレームワークが選択され、技術スタック決定に文書化されている
- フレームワークがプロジェクトの依存関係に含まれている（package.json、pom.xml、requirements.txt など）
- フレームワークがカスタムジェネレーター、縮小、シードベースの再現性をサポートする
- プロジェクトが複数の言語を使用する場合、PBT 適用可能なコードを持つ各言語にフレームワークが選択されている

---

## ルール PBT-10: 補完的テスト戦略

**ルール**: プロパティベーステストは例ベーステストを補完しなければならず、置き換えてはなりません。2 つのアプローチは異なる目的を果たします:

- **例ベーステスト**: 特定の既知のシナリオ、回帰ケース、および明示的な期待値を持つビジネスクリティカルなエッジケースを文書化します。具体的な動作の実行可能なドキュメントとして機能します。
- **プロパティベーステスト**: 広い入力空間にまたがる一般的な不変条件を検証します。未知のエッジケースを発見し、プロパティが普遍的に成立することを検証します。

要件:
- ユーザーストーリーまたは要件で特定されたクリティカルなビジネスシナリオは、PBT が同じプロパティをカバーしていても、明示的な例ベーステストを持たなければならない
- PBT はビジネスクリティカルなパスの唯一のテストであってはならない — 主要なシナリオの期待される動作を固定するために、少なくとも 1 つの例ベーステストが必要
- PBT が失敗ケースを発見した場合、縮小された最小限の例を永続的な例ベースの回帰テストとして追加すべきである
- テストドキュメントは例ベーステストとプロパティベーステストを明確に区別しなければならない（別々のテストファイル、テストクラス、または明確に名付けられたテスト関数）

**検証**:
- ビジネスクリティカルなパスに例ベーステストとプロパティベーステストの両方がある
- 重要な機能のテストカバレッジとして PBT のみが使用されない
- テストファイルまたはテストクラスが PBT と例ベーステストを明確に分離またはラベル付けする
- PBT で発見された失敗からの回帰テストが永続的な例ベーステストとして取り込まれている

---

## 強制適用の統合

これらのルールは、以下の AI-DLC ステージに適用される横断的制約です:

| ステージ | 適用されるルール | 強制適用 |
|---|---|---|
| 機能設計 | PBT-01 | プロパティ特定が設計アーティファクトに表れなければならない |
| 非機能要件 | PBT-09 | フレームワーク選択が技術スタック決定に含まれなければならない |
| コード生成（計画） | PBT-01 〜 PBT-10 | コード生成計画に特定されたプロパティの PBT テストステップが含まれなければならない |
| コード生成（生成） | PBT-02 〜 PBT-08、PBT-10 | 生成されたテストに例ベーステストと並んで PBT が含まれなければならない |
| ビルドとテスト | PBT-08 | テスト実行手順にシードロギングと CI 統合を含む PBT が含まれなければならない |

各該当ステージで:
- 生成されたアーティファクトに対してすべての PBT ルール検証基準を評価する
- ステージ完了サマリーに各ルールの準拠、非準拠、または N/A をリストした「PBT Compliance」セクションを含める
- ルールが非準拠の場合、それはブロッキング PBT 発見です — 概要で定義されたブロッキング発見の動作に従う
- 設計ドキュメントとテスト手順に PBT ルール参照を含める

## 付録: プロパティカテゴリクイックリファレンス

機能設計時にプロパティを特定する開発者と AI モデル向け（PBT-01）:

| パターン名 | 正式な用語 | テストの形 | 使用タイミング |
|---|---|---|---|
| 行って帰る | 可逆関数 | `f_inv(f(x)) == x` | シリアライゼーション、エンコーディング、パース |
| 変わるものは何もない | 不変条件 | `measure(f(x)) == measure(x)` | ソート、map、filter、変換 |
| 変わればいつも変わらず | 冪等性 | `f(f(x)) == f(x)` | 正規化、重複排除、キャッシュ書き込み |
| 別の道、同じ目的地 | 可換性 | `f(g(x)) == g(f(x))` | 算術、集合演算、独立した変換 |
| まず小さな問題を解く | 構造的帰納法 | `x` の性質が `x + 要素` に引き継がれる | 再帰的構造、リスト、木 |
| 証明は難しく、確認は容易 | 検証 | `verify(solve(x)) == true` | ソルバー、最適化、検索アルゴリズム |
| テストオラクル | 参照比較 | `f(x) == oracle(x)` | 最適化対ブルートフォース、リファクタリング対レガシー |
