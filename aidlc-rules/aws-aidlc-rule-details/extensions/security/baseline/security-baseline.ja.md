# ベースラインセキュリティルール

## 概要
これらのセキュリティルールは、すべての AI-DLC フェーズにまたがって適用される**必須**の横断的制約です。任意のガイダンスではなく、質問生成・設計アーティファクト作成・コード生成・完了メッセージ表示の各ステージが**強制適用しなければならない**ハード制約です。

**強制適用**: 各該当ステージにおいて、モデルはステージ完了メッセージをユーザーに提示する前に、これらのルールへの準拠を検証しなければなりません。

### セキュリティ発見のブロッキング動作
**ブロッキングセキュリティ発見**とは:
1. 発見内容は SECURITY ルール ID と説明を含む「Security Findings」セクションとしてステージ完了メッセージに記載しなければならない
2. すべてのブロッキング発見が解消されるまで、ステージは「次のステージへ進む」オプションを提示してはならない
3. モデルは「変更を要求」オプションのみを提示し、変更が必要な内容を明確に説明しなければならない
4. 発見内容は SECURITY ルール ID、説明、ステージのコンテキストとともに `aidlc-docs/audit.md` に記録しなければならない

SECURITY ルールが現在のプロジェクトに適用されない場合（例: データストアが存在しない場合の SECURITY-01）は、準拠サマリーで **N/A** とマークしてください — これはブロッキング発見ではありません。

### デフォルトの強制適用
このドキュメントのすべてのルールはデフォルトで**ブロッキング**です。ルールの検証基準が満たされない場合、それはブロッキングセキュリティ発見です — 上記で定義されたブロッキング発見の動作に従ってください。

### 検証基準のフォーマット
このドキュメントの検証項目は、準拠チェックを説明するプレーンな箇条書きです。ステージ計画ファイルで使用される `- [ ]` / `- [x]` の進捗追跡チェックボックスとは区別されます。各項目はレビュー時に準拠または非準拠として評価してください。

## ルール SECURITY-01: 保存時と転送時の暗号化

**ルール**: すべてのデータ永続化ストア（データベース、オブジェクトストレージ、ファイルシステム、キャッシュ、またはそれに相当するもの）には以下が必要です:
- マネージドキーサービスまたはカスタマーマネージドキーを使用した保存時暗号化が有効であること
- 転送時暗号化が強制されていること（ストアとの間のすべてのデータ移動に TLS 1.2 以上を使用）

**検証**:
- 暗号化構成ブロックなしにストレージリソースが定義されていない
- 暗号化されていないプロトコルを使用するデータベース接続文字列がない
- オブジェクトストレージが保存時暗号化を強制し、ポリシーによって非 TLS リクエストを拒否する
- データベースインスタンスでストレージ暗号化が有効化され、TLS 接続が強制されている

---

## ルール SECURITY-02: ネットワーク仲介コンポーネントでのアクセスロギング

**ルール**: 外部トラフィックを処理するすべてのネットワーク対応仲介コンポーネントでアクセスロギングを有効にしなければなりません。これには以下が含まれます:
- ロードバランサー → 永続ストアへのアクセスログ
- API ゲートウェイ → 集中ログサービスへの実行ログとアクセスログ
- CDN ディストリビューション → 標準ログまたはリアルタイムログ

**検証**:
- アクセスロギングを有効にせずにロードバランサーリソースが定義されていない
- アクセスロギングを設定せずに API ゲートウェイステージが定義されていない
- ロギング設定なしに CDN ディストリビューションが定義されていない

---

## ルール SECURITY-03: アプリケーションレベルのロギング

**ルール**: すべてのデプロイされたアプリケーションコンポーネントには構造化ロギングインフラストラクチャが必要です:
- ロギングフレームワークが設定されていること
- ログ出力が集中ログサービスに向けられていること
- ログにはタイムスタンプ、相関/リクエスト ID、ログレベル、メッセージが含まれること
- 機密データ（パスワード、トークン、PII）をログ出力に含めないこと

**検証**:
- すべてのサービス/関数エントリポイントに設定済みロガーが含まれている
- 本番コードの主要なロギングメカニズムとして場当たり的なログ出力が使用されていない
- ログ設定が出力を集中ログサービスにルーティングする
- シークレット、トークン、PII がログに記録されていない

---

## ルール SECURITY-04: Web アプリケーション向け HTTP セキュリティヘッダー

**ルール**: HTML を配信するすべてのエンドポイントで、以下の HTTP レスポンスヘッダーを設定しなければなりません:

| ヘッダー | 必要な値 |
|---|---|
| `Content-Security-Policy` | 制限的なポリシーを定義する（最低限: `default-src 'self'`）|
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` |
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY`（フレーミングが必要な場合は `SAMEORIGIN`）|
| `Referrer-Policy` | `strict-origin-when-cross-origin` |

**注記**: `X-XSS-Protection` は現代のブラウザでは非推奨です。代わりに `Content-Security-Policy` を使用してください。

**検証**:
- ミドルウェアまたはレスポンスインターセプターがすべての必要なヘッダーを設定する
- CSP ポリシーが文書化された正当な理由なく `unsafe-inline` や `unsafe-eval` を使用しない
- HSTS max-age が少なくとも 31536000（1 年）である

---

## ルール SECURITY-05: すべての API パラメーターの入力バリデーション

**ルール**: すべての API エンドポイント（REST、GraphQL、gRPC、WebSocket）は処理前にすべての入力パラメーターを検証しなければなりません。バリデーションには以下を含める必要があります:
- **型チェック**: 予期しない型を拒否する
- **長さ/サイズ制限**: 文字列の最大長、配列とペイロードの最大サイズを強制する
- **フォーマットバリデーション**: 構造化された入力（メール、日付、ID）に対してアローリスト（正規表現またはスキーマ）を使用する
- **サニタイゼーション**: XSS を防ぐためにユーザー提供の文字列内の HTML/スクリプトコンテンツをエスケープまたは拒否する
- **インジェクション防止**: すべてのデータベース操作にパラメータ化クエリを使用する（文字列連結は不可）

**検証**:
- すべての API ハンドラーがバリデーションライブラリまたはスキーマを使用する
- 生のユーザー入力が SQL、NoSQL、または OS コマンドに連結されていない
- 文字列入力に明示的な最大長制約がある
- リクエストボディサイズ制限がフレームワークまたはゲートウェイレベルで設定されている

---

## ルール SECURITY-06: 最小権限アクセスポリシー

**ルール**: すべての ID およびアクセス管理ポリシー、ロール、または権限境界は最小権限に従わなければなりません:
- 特定のリソース識別子を使用する — API がリソースレベルの権限をサポートしない場合を除き、ワイルドカードリソースは**絶対に使用しない**（例外を文書化する）
- 特定のアクションを使用する — ワイルドカードアクションは**絶対に使用しない**
- 可能な場合は条件をスコープする
- 読み取りと書き込みの権限を別々のポリシーステートメントに分離する

**検証**:
- 文書化された例外なしに、ポリシーにワイルドカードアクションやワイルドカードリソースが含まれていない
- サービスロールがそのサービスが実際に呼び出す内容より広い権限を持っていない
- 可能な場合、インラインポリシーの代わりにマネージドポリシーが使用されている
- すべてのロールに特定のサービスまたはアカウントにスコープされた信頼ポリシーがある

---

## ルール SECURITY-07: 制限的なネットワーク設定

**ルール**: すべてのネットワーク設定（セキュリティグループ、ネットワーク ACL、ルートテーブル）はデフォルト拒否に従わなければなりません:
- ファイアウォールルール: アプリケーションが必要とする特定のポートのみを開放する
- ポート 80/443 のパブリック向けロードバランサーを除き、ソースが `0.0.0.0/0` のインバウンドルールなし
- 明示的に正当化されない限り、すべてのポートに `0.0.0.0/0` のアウトバウンドルールなし
- プライベートサブネットには直接インターネットゲートウェイルールを設定してはならない
- 利用可能な場合はクラウドサービスアクセスにプライベートエンドポイントを使用する

**検証**:
- パブリックロードバランサーのポート 80/443 以外のポートで `0.0.0.0/0` のインバウンドを許可するファイアウォールルールがない
- データベースとアプリケーションのファイアウォールルールが特定の CIDR ブロックまたはセキュリティグループ参照にソースを制限する
- プライベートサブネットが NAT ゲートウェイ（インターネットゲートウェイではなく）を経由してルーティングする
- 高トラフィックのクラウドサービス呼び出しにプライベートエンドポイントが使用されている

---

## ルール SECURITY-08: アプリケーションレベルのアクセス制御

**ルール**: リソースにアクセスまたは変更するすべてのアプリケーションエンドポイントは、アプリケーション層での認可チェックを強制しなければなりません:
- **デフォルト拒否**: 明示的にパブリックとマークされない限り、すべてのルート/エンドポイントは認証を必要とする
- **オブジェクトレベルの認可**: ID によってリソースを参照するすべてのリクエストは、リクエスト元のユーザー/プリンシパルがそのリソースを所有または アクセスする権限があることを検証しなければならない（IDOR の防止）
- **機能レベルの認可**: 管理者または特権操作は、サーバーサイドで呼び出し元のロール/権限をチェックしなければならない — クライアントサイドの非表示に依存しないこと
- **CORS ポリシー**: クロスオリジンリソース共有は明示的に許可されたオリジンのみに制限しなければならない — 認証されたエンドポイントで `Access-Control-Allow-Origin: *` は絶対に使用しない
- **トークンバリデーション**: JWT またはセッショントークンは、すべてのリクエストでサーバーサイドで検証しなければならない（署名、有効期限、オーディエンス、発行者）

**検証**:
- すべてのコントローラー/ハンドラーに認可ミドルウェアまたはガードが適用されている
- リソース ID のデータを返すエンドポイントが、呼び出し元の所有権または権限を検証せずにデータを返さない
- 管理者/特権ルートにサーバーサイドで強制された明示的なロールチェックがある
- CORS 設定が認証されたエンドポイントでワイルドカードオリジンを使用しない
- トークンバリデーションはすべてのリクエストでサーバーサイドで実行される（ログイン時だけでなく）

---

## ルール SECURITY-09: セキュリティ強化と設定ミスの防止

**ルール**: すべてのデプロイされたコンポーネントはハードニングベースラインに従わなければなりません:
- **デフォルト認証情報なし**: デプロイ前にデフォルトのユーザー名/パスワードを変更または無効化しなければならない
- **最小限のインストール**: 未使用のフィーチャー、フレームワーク、サンプルアプリケーション、ドキュメントエンドポイントを削除または無効化する
- **エラー処理**: 本番環境のエラーレスポンスはエンドユーザーにスタックトレース、内部パス、フレームワークバージョン、データベースの詳細を公開してはならない
- **ディレクトリリスティング**: Web サーバーはディレクトリリスティングを無効化しなければならない
- **クラウドストレージ**: 明示的に必要とされ文書化されていない限り、クラウドオブジェクトストレージはパブリックアクセスをブロックしなければならない
- **パッチ管理**: ランタイム環境、フレームワーク、OS イメージは現在サポートされているバージョンを使用しなければならない

**検証**:
- 設定ファイル、環境変数、IaC テンプレートにデフォルト認証情報が存在しない
- 本番環境のエラーレスポンスが汎用的なメッセージを返す（スタックトレースや内部詳細なし）
- 文書化された例外がない限り、クラウドオブジェクトストレージがパブリックアクセスをブロックしている
- サンプル/デモアプリケーションやデフォルトページがデプロイされていない
- フレームワークとランタイムのバージョンが現在サポートされている

---

## ルール SECURITY-10: ソフトウェアサプライチェーンセキュリティ

**ルール**: すべてのプロジェクトはソフトウェアサプライチェーンを管理しなければなりません:
- **依存関係のピン留め**: すべての依存関係は正確なバージョンまたはロックファイルを使用しなければならない
- **脆弱性スキャン**: 依存関係の脆弱性スキャナーを設定しなければならない
- **未使用依存関係なし**: アクティブに使用されていないパッケージを削除する
- **信頼されたソースのみ**: 依存関係は公式レジストリまたは検証済みプライベートレジストリから取得しなければならない — 確認されていないサードパーティソースは使用不可
- **SBOM**: プロジェクトは本番デプロイメント向けにソフトウェア部品表（SBOM）を生成しなければならない
- **CI/CD 整合性**: ビルドパイプラインはピン留めされたツールバージョンと検証済みベースイメージを使用しなければならない — 本番 Dockerfile や CI 設定で `latest` タグは使用不可

**検証**:
- ロックファイルが存在し、バージョン管理にコミットされている
- 依存関係の脆弱性スキャンステップが CI/CD に含まれているか、ビルド手順に文書化されている
- 未使用または放棄された依存関係が含まれていない
- Dockerfile と CI 設定が本番向けに `latest` またはピン留めされていないイメージタグを使用しない
- 依存関係が公式または検証済みレジストリから取得されている

---

## ルール SECURITY-11: セキュアな設計原則

**ルール**: アプリケーション設計は最初からセキュリティを組み込まなければなりません:
- **関心事の分離**: セキュリティクリティカルなロジック（認証、認可、支払い処理）は専用モジュールに分離しなければならない — コードベース全体に散らばらせてはならない
- **多層防御**: 単一のコントロールだけが唯一の防御ラインであってはならない — コントロールを重ねる（バリデーション + 認可 + 暗号化）
- **レート制限**: パブリック向けエンドポイントは悪用を防ぐためにレート制限またはスロットリングを実装しなければならない
- **ビジネスロジックの悪用**: 設計はハッピーパスシナリオだけでなく、悪用のケースも考慮しなければならない

**検証**:
- セキュリティクリティカルなロジックが専用モジュールまたはサービスにカプセル化されている
- パブリック向け API でレート制限が設定されている
- 設計ドキュメントが少なくとも 1 つの悪用/乱用シナリオに対処している

---

## ルール SECURITY-12: 認証とクレデンシャル管理

**ルール**: ユーザー認証があるすべてのアプリケーションは以下を実装しなければなりません:
- **パスワードポリシー**: 最低 8 文字、漏洩パスワードリストとの照合
- **クレデンシャルストレージ**: パスワードはアダプティブアルゴリズムを使用してハッシュ化しなければならない — 弱いまたは非アダプティブなハッシュ化は不可
- **多要素認証**: MFA は管理者アカウントでサポートされなければならず、すべてのユーザーに対しても利用可能にすべきである
- **セッション管理**: セッションはサーバーサイドの有効期限を持ち、ログアウト時に無効化され、secure/httpOnly/sameSite Cookie 属性を使用しなければならない
- **ブルートフォース保護**: ログインエンドポイントは繰り返し失敗後にアカウントロックアウト、段階的な遅延、または CAPTCHA を実装しなければならない
- **ハードコードされた認証情報なし**: ソースコードや IaC テンプレートにパスワード、API キー、シークレットを含めない — シークレットマネージャーを使用する

**検証**:
- パスワードハッシュがアダプティブアルゴリズムを使用する（弱いまたは非アダプティブなハッシュ化ではない）
- セッション Cookie に `Secure`、`HttpOnly`、`SameSite` 属性が設定されている
- ログインエンドポイントにブルートフォース保護がある（ロックアウト、遅延、または CAPTCHA）
- ソースコードや設定ファイルにハードコードされた認証情報がない
- MFA が管理者アカウントでサポートされている
- セッションはログアウト時に無効化され、定義された有効期限がある

---

## ルール SECURITY-13: ソフトウェアとデータの整合性検証

**ルール**: システムはソフトウェアとデータの整合性を検証しなければなりません:
- **デシリアライゼーションの安全性**: 信頼されていないデータはバリデーションなしにデシリアライズしてはならない — 安全なデシリアライゼーションライブラリまたは許可された型のアローリストを使用する
- **アーティファクトの整合性**: ダウンロードされた依存関係、プラグイン、アップデートはチェックサムまたはデジタル署名で検証しなければならない
- **CI/CD パイプラインセキュリティ**: ビルドパイプラインはパイプライン定義を変更できる人を制限しなければならない — コード作成者とデプロイ承認者の職務を分離する
- **CDN と外部リソース**: 外部 CDN から読み込まれるスクリプトやリソースはサブリソース整合性（SRI）ハッシュを使用しなければならない
- **データ整合性**: 重要なデータの変更は監査可能でなければならない（誰が何をいつ変更したか）

**検証**:
- 信頼されていない入力の安全でないデシリアライゼーションがない
- CDN から読み込まれる外部スクリプトに SRI 整合性属性が含まれている
- CI/CD パイプライン定義がアクセス制御されており、変更が監査可能である
- 重要なデータ変更がアクター、タイムスタンプ、変更前後の値とともにログに記録されている

---

## ルール SECURITY-14: アラートとモニタリング

**ルール**: ロギング（SECURITY-02、SECURITY-03）に加えて、システムには以下が必要です:
- **セキュリティイベントアラート**: 高価値のセキュリティイベントに対してアラートを設定しなければならない: 繰り返しの認証失敗、権限昇格の試み、異常な場所からのアクセス、認可失敗
- **ログの整合性**: ログは追記専用または改ざん防止ストレージに保存しなければならない — アプリケーションコードは自身の監査ログを削除または変更できてはならない
- **ログ保持**: ログはアプリケーションのコンプライアンス要件に適した最小期間保持しなければならない（デフォルト: 最低 90 日）
- **モニタリングダッシュボード**: 主要な運用およびセキュリティメトリクスに対してモニタリングダッシュボードまたはアラーム設定を定義しなければならない

**検証**:
- 認証失敗と認可違反に対してアラートが設定されている
- アプリケーションロググループに保持ポリシーが設定されている（最低 90 日）
- アプリケーションロールが自身のロググループ/ストリームを削除する権限を持っていない
- セキュリティ関連イベント（ログイン失敗、アクセス拒否、権限変更）がアラートを生成する

## ルール SECURITY-15: 例外処理とフェイルセーフデフォルト

**ルール**: すべてのアプリケーションは例外的な状況を安全に処理しなければなりません:
- **キャッチと処理**: すべての外部呼び出し（データベース、API、ファイル I/O）は明示的なエラー処理を持たなければならない — 本番環境でのハンドルされていない Promise の拒否や未捕捉の例外なし
- **フェイルクローズ**: エラー時に、システムはアクセスを拒否するか操作を停止しなければならない — フェイルオープンは不可
- **リソースのクリーンアップ**: エラーパスはリソース（接続、ファイルハンドル、ロック）を解放しなければならない — try/finally、using ステートメント、または同等のパターンを使用する
- **ユーザー向けエラー**: ユーザーに表示されるエラーメッセージは汎用的でなければならない — 内部詳細やシステム情報を含まない
- **グローバルエラーハンドラー**: アプリケーションは未処理の例外をキャッチし、（SECURITY-03 に従って）ログに記録し、安全なレスポンスを返すグローバル/トップレベルのエラーハンドラーを持たなければならない

**検証**:
- すべての外部呼び出し（DB、HTTP、ファイル I/O）に明示的なエラー処理がある（try/catch、.catch()、エラーコールバック）
- アプリケーションエントリポイントにグローバルエラーハンドラーが設定されている
- エラーパスが認可やバリデーションチェックをバイパスしない（フェイルクローズ）
- リソースがエラーパスでクリーンアップされる（接続のクローズ、トランザクションのロールバック）
- アプリケーションコードでハンドルされていない Promise の拒否や未捕捉の例外警告がない

## 強制適用の統合

これらのルールは、すべての AI-DLC ステージに適用される横断的制約です。各ステージで:
- 生成されたアーティファクトに対してすべての SECURITY ルール検証基準を評価する
- ステージ完了サマリーに各ルールの準拠、非準拠、または N/A をリストした「Security Compliance」セクションを含める
- ルールが非準拠の場合、それはブロッキングセキュリティ発見です — 概要で定義されたブロッキング発見の動作に従う
- 設計ドキュメントとテスト手順にセキュリティルール参照を含める
