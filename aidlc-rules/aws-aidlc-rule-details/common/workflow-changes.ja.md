# ワークフロー途中での変更とステージ管理

## 概要

ユーザーはワークフロー実行中に実行計画やステージの実行に対する変更をリクエストする場合があります。このドキュメントでは、これらのリクエストを安全かつ効果的に処理するためのガイダンスを提供します。

---

## ワークフロー途中での変更の種類

### 1. スキップされたステージの追加

**シナリオ**: ユーザーが元々スキップされたステージを追加したい

**例**: 「やっぱり、そのステージはスキップしたけれどユーザーストーリーを追加したい」

**対応方法**:
1. **リクエストを確認する**: 「ユーザーストーリーステージを追加しますね。これによりユーザーストーリーとペルソナが作成されます。確認しますか？」
2. **依存関係を確認する**: 前提となるすべてのステージが完了していることを確認する
3. **実行計画を更新する**: 根拠とともに `execution-plan.md` にステージを追加する
4. **状態を更新する**: `aidlc-state.md` でステージを「PENDING」としてマークする
5. **ステージを実行する**: 通常のステージ実行プロセスに従う
6. **変更を記録する**: タイムスタンプと理由とともに `audit.md` に記録する

**考慮事項**:
- 新しいアーティファクトから恩恵を受ける可能性のある後のステージを更新する必要がある場合がある
- 新しい情報を取り込むために既存のアーティファクトの修正が必要な場合がある
- タイムラインが延長される

---

### 2. 計画済みステージのスキップ

**シナリオ**: ユーザーが実行予定のステージをスキップしたい

**例**: 「今は非機能要件設計ステージをスキップしましょう」

**対応方法**:
1. **リクエストを確認する**: 「非機能要件設計をスキップしますね。これにより非機能要件パターンや論理コンポーネントは組み込まれません。確認しますか？」
2. **影響について警告する**: 欠如するものと潜在的な影響を説明する
3. **明示的な確認を得る**: ユーザーは影響を理解していることを明示的に確認する必要がある
4. **実行計画を更新する**: 理由とともにステージを「SKIPPED」としてマークする
5. **状態を更新する**: `aidlc-state.md` でステージを「SKIPPED」としてマークする
6. **後のステージを調整する**: 後のステージで手動設定が必要な場合があることをメモする
7. **変更を記録する**: タイムスタンプと理由とともに `audit.md` に記録する

**考慮事項**:
- 後のステージが失敗したり手動での介入が必要になったりする場合がある
- ユーザーが欠けているアーティファクトの責任を受け入れる
- 必要に応じて後で追加することができる

---

### 3. 現在のステージの再実行

**シナリオ**: ユーザーが現在のステージの結果に不満でやり直したい

**例**: 「このユーザーストーリーが気に入りません。最初からやり直せますか？」

**対応方法**:
1. **懸念事項を理解する**: 「ストーリーの何を変えたいですか？」
2. **選択肢を提示する**:
   - **オプション A**: 既存のアーティファクトを修正する（速い、一部の作業を保持）
   - **オプション B**: 完全に最初からやり直す（白紙、より時間がかかる）
3. **やり直しを選択した場合**:
   - 既存のアーティファクトをアーカイブする: `{artifact}.backup.{timestamp}`
   - 計画ファイル内のステージチェックボックスをリセットする
   - `aidlc-state.md` でステージを「IN PROGRESS」としてマークする
   - ステージの完了ステータスをクリアする
   - 最初から再実行する
4. **変更を記録する**: やり直しの理由と変更される内容を記録する

**考慮事項**:
- 既存の作業は失われる（ただしバックアップされる）
- 依存するステージを再実行する必要がある場合がある
- タイムラインが延長される

---

### 4. 以前のステージの再実行

**シナリオ**: ユーザーが完了済みのステージに戻ってやり直したい

**例**: 「以前に行ったアーキテクチャの決定を変更したい」

**対応方法**:
1. **影響を評価する**: 再実行するステージに依存するすべてのステージを特定する
2. **ユーザーに警告する**: 「アプリケーション設計を再実行すると、以下を再実行する必要があります: ユニット計画、ユニット生成、ユニットごとの設計（すべてのユニット）、コード生成。確認しますか？」
3. **明示的な確認を得る**: ユーザーは全影響を理解する必要がある
4. **確認された場合**:
   - 影響を受けるすべてのアーティファクトをアーカイブする
   - `aidlc-state.md` で影響を受けるすべてのステージをリセットする
   - 影響を受けるすべての計画ファイルのチェックボックスをクリアする
   - 再実行するステージに戻る
   - その時点から順に再実行する
5. **変更を記録する**: 完全な影響と再実行の理由を記録する

**考慮事項**:
- 大幅な再作業が必要
- すべての依存するステージを再実行する必要がある
- タイムラインが大幅に延長される
- 再起動よりも修正の方が良い場合があるかどうかを検討する

---

### 5. ステージの詳細度の変更

**シナリオ**: ユーザーが現在または今後のステージの詳細度レベルを変更したい

**例**: 「標準ではなく包括的な要件分析を行いましょう」

**対応方法**:
1. **リクエストを確認する**: 「要件分析を標準から包括的な深度に変更しますね。より徹底的になりますがより時間がかかります。確認しますか？」
2. **実行計画を更新する**: `workflow-planning.md` の深度レベルを変更する
3. **アプローチを調整する**: そのステージの包括的な深度ガイドラインに従う
4. **見積もりを更新する**: 新しいタイムライン見積もりをユーザーに伝える
5. **変更を記録する**: 深度の変更と理由を記録する

**考慮事項**:
- 深度が高い = 時間はかかるが品質が高い
- 深度が低い = 速いが詳細が欠ける場合がある
- ステージの前または実行中にのみ変更可能。完了後は変更不可

---

### 6. ワークフローの一時停止

**シナリオ**: ユーザーが一時停止して後で再開する必要がある

**例**: 「今日は止める必要があり、明日続けます」

**対応方法**:
1. **現在のステップを完了する**: 可能であれば進行中の現在のステップを終わらせる
2. **チェックボックスを更新する**: 完了したすべてのステップを [x] でマークする
3. **状態を更新する**: `aidlc-state.md` が現在のステータスを反映していることを確認する
4. **一時停止を記録する**: `audit.md` に一時停止ポイントを記録する
5. **再開手順を提供する**: 「戻ってきたら、既存のプロジェクトを検出して以下から続けることを提案します: [現在のステージ、現在のステップ]」

**再開時**:
1. **既存プロジェクトを検出する**: `aidlc-state.md` の有無を確認する
2. **コンテキストを読み込む**: 完了したステージのすべてのアーティファクトを読み込む
3. **ステータスを表示する**: 現在のステージと次のステップを表示する
4. **選択肢を提示する**: 前回の場所から続けるか前の作業をレビューするか
5. **再開を記録する**: `audit.md` に再開ポイントを記録する

---

### 7. アーキテクチャの決定変更

**シナリオ**: ユーザーがモノリスからマイクロサービスに変更したい（またはその逆）

**例**: 「実はモノリスではなくマイクロサービスにしましょう」

**対応方法**:
1. **現在の進捗を評価する**: ワークフローのどこまで進んでいるかを確認する
2. **影響を説明する**:
   - ユニット計画前の場合: 影響は最小限 — 決定を更新するだけ
   - ユニット計画後の場合: ユニット計画、ユニット生成、すべてのユニットごとの設計を再実行する必要がある
   - コード生成後の場合: 大幅な再作業が必要
3. **アプローチを推奨する**:
   - ワークフローの早い段階: アプリケーション設計ステージから再開する
   - ワークフローの遅い段階: 修正が可能かどうか vs. 再開コストを検討する
4. **確認を得る**: ユーザーは変更の全範囲を理解する必要がある
5. **変更を実行する**: 影響を受けるステージの再実行手順に従う

**考慮事項**:
- アーキテクチャの変更は連鎖的な影響をもたらす
- ワークフローの早い段階 = 変更が容易
- ワークフローの遅い段階 = コストと利益を検討する

---

### 8. ユニットの追加/削除

**シナリオ**: ユニット生成後にユニットを追加または削除したい

**例**: 「Payment ユニットを Payment と Billing に分割する必要がある」

**対応方法**:
1. **影響を評価する**: どのユニットが設計/コードを完了したかを確認する
2. **影響を説明する**:
   - ユニットの追加: 新しいユニットの完全な設計とコードが必要
   - ユニットの削除: 機能を他のユニットに再配分する必要がある
   - ユニットの分割: 両方の結果ユニットの設計とコードを再実行する必要がある
3. **ユニットアーティファクトを更新する**:
   - `unit-of-work.md` を修正する
   - `unit-of-work-dependency.md` を更新する
   - `unit-of-work-story-map.md` を修正する
4. **影響を受けるユニットをリセットする**: 影響を受けるユニットを再設計が必要としてマークする
5. **変更を実行する**: 影響を受けるユニットに対して通常のユニット設計とコードのプロセスに従う

**考慮事項**:
- それらのユニットのすべての下流ステージに影響する
- 依存関係が変わると他のユニットに影響する可能性がある
- タイムラインへの影響は影響を受けるユニットの数による

---

## 変更処理の一般的なガイドライン

### 変更前

1. **リクエストを理解する**: ユーザーが何を変えたいかとその理由について確認の質問をする
2. **影響を評価する**: 影響を受けるすべてのステージ、アーティファクト、依存関係を特定する
3. **影響を説明する**: 再実行が必要なことと時間的影響を明確に伝える
4. **代替手段を提案する**: 再開よりも修正の方が良い場合がある
5. **明示的な確認を得る**: ユーザーは影響を理解し了承する必要がある

### 変更中

1. **既存の作業をアーカイブする**: 破壊的な変更の前には必ずバックアップする
2. **すべての追跡を更新する**: `aidlc-state.md`、計画ファイル、`audit.md` を同期して保つ
3. **進捗を伝える**: 何が起きているかをユーザーに伝え続ける
4. **変更を検証する**: 変更がすべてのアーティファクト間で一貫していることを確認する
5. **継続性をテストする**: 変更後もワークフローが滑らかに続けられることを確認する

### 変更後

1. **一貫性を確認する**: すべてのアーティファクトが変更と一致していることを確認する
2. **ドキュメントを更新する**: すべての参照が更新されていることを確認する
3. **完全に記録する**: `audit.md` に完全な変更履歴を記録する
4. **ユーザーに確認する**: 変更がユーザーの期待を満たしていることを確認する
5. **ワークフローを再開する**: 新しい状態から通常の実行を継続する

## 変更リクエストのデシジョンツリー

```
ユーザーが変更をリクエストする
    |
    ├─ 現在のステージですか？
    |   ├─ はい: 現在のステージを修正または再実行できる
    |   └─ いいえ: 次の質問へ
    |
    ├─ 完了済みのステージですか？
    |   ├─ はい: 依存するステージへの影響を評価する
    |   |   ├─ 影響が小さい: 修正して依存関係を更新する
    |   |   └─ 影響が大きい: そのステージからの再開を推奨する
    |   └─ いいえ: 次の質問へ
    |
    ├─ スキップされたステージの追加ですか？
    |   ├─ はい: 前提条件を確認し、計画に追加し、実行する
    |   └─ いいえ: 次の質問へ
    |
    ├─ 計画済みステージのスキップですか？
    |   ├─ はい: 影響について警告し、確認を得てスキップする
    |   └─ いいえ: 次の質問へ
    |
    └─ 詳細度レベルの変更ですか？
        ├─ はい: 計画を更新し、アプローチを調整する
        └─ いいえ: ユーザーにリクエストを明確化してもらう
```

## ログ要件

### 変更リクエストのログフォーマット

```markdown
## Change Request - [ステージ名]
**Timestamp**: [ISO タイムスタンプ]
**Request**: [ユーザーが変更したいこと]
**Current State**: [ワークフローの現在地]
**Impact Assessment**: [影響を受けること]
**User Confirmation**: [ユーザーの明示的な確認]
**Action Taken**: [行った対応]
**Artifacts Affected**: [変更/リセットされたファイルのリスト]

---
```

## ベストプラクティス

1. **常に確認する**: 明示的なユーザーの確認なく破壊的な変更を行わない
2. **影響を説明する**: ユーザーは決定前に影響を理解する必要がある
3. **選択肢を提示する**: 変更を処理する方法が複数ある場合がある
