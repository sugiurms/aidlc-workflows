# 適応的な詳細度

**目的**: AI-DLC が問題の複雑さに応じて詳細レベルをどのように適応させるかを説明する

## 基本原則

**ステージが実行されると、定義されているすべてのアーティファクトが作成されます。「詳細度」とは、それらのアーティファクト内の詳細と厳密さのレベルを指し、問題の複雑さに応じて適応します。**

## ステージ選択 vs 詳細レベル

### ステージ選択（二択）
- **ワークフロー計画**が決定する: 各ステージを「実行する (EXECUTE)」か「スキップする (SKIP)」か
- **実行 (EXECUTE) の場合**: ステージが実行され、定義されているすべてのアーティファクトが作成される
- **スキップ (SKIP) の場合**: ステージはまったく実行されない

### 詳細レベル（適応的）
- **単純な問題**: 必要最低限の詳細を含む簡潔なアーティファクト
- **複雑な問題**: 詳細な内容を含む包括的なアーティファクト
- **モデルが判断する**: 規定のルールではなく、問題の特性に基づいて決定する

## 詳細レベルに影響する要因

モデルは適切な詳細度を決定する際に以下の要因を考慮します:

1. **要求の明確さ**: ユーザーの要求はどれほど明確で完全か？
2. **問題の複雑さ**: 解決策の空間はどれほど複雑か？
3. **スコープ**: 単一ファイル、コンポーネント、複数コンポーネント、またはシステム全体か？
4. **リスクレベル**: エラーや欠如の影響はどの程度か？
5. **利用可能なコンテキスト**: グリーンフィールドかブラウンフィールドか、既存のドキュメントの有無
6. **ユーザーの好み**: ユーザーが簡潔さや詳細さについて意向を示しているか？

## 例: 要件分析のアーティファクト

**すべてのシナリオで同じアーティファクトが作成されます**:
- `requirement-verification-questions.md`（必要な場合）
- `requirements.md`

**注記**: ユーザーの最初のリクエストは `audit.md` に記録されます（別途 user-intent.md は不要）。

**詳細レベルは複雑さによって異なります**:

### 単純なシナリオ（バグ修正）
- **requirement-verification-questions.md**: 必要な確認事項の質問
- **requirements.md**: 簡潔な機能要件、最小限のセクション

### 複雑なシナリオ（システム移行）
- **requirement-verification-questions.md**: 複数ラウンド、10 件以上の質問
- **requirements.md**: 包括的な機能・非機能要件、トレーサビリティ、受け入れ基準

## 例: アプリケーション設計のアーティファクト

**すべてのシナリオで同じアーティファクトが作成されます**:
- `application-design.md`
- `component-diagram.md`

**詳細レベルは複雑さによって異なります**:

### 単純なシナリオ（単一コンポーネント）
- **application-design.md**: 基本的なコンポーネントの説明、主要メソッド
- **component-diagram.md**: 主要な関係を示すシンプルな図

### 複雑なシナリオ（マルチコンポーネントシステム）
- **application-design.md**: 詳細なコンポーネントの責務、シグネチャ付きのすべてのメソッド、設計パターン、検討した代替案
- **component-diagram.md**: すべての関係、データフロー、統合ポイントを含む包括的な図

## モデルへの指針

**「手元の問題に必要な詳細だけを作成する — それ以上でもそれ以下でもなく。」**

- 不必要な詳細で単純な問題を人為的に膨らませない
- 重要な詳細を省略して複雑な問題を不十分に扱わない
- 問題の特性が自然に詳細レベルを決定するようにする
- ステージが実行される際は、必要なアーティファクトが常に作成される
