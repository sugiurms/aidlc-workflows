# 質問フォーマットガイド

## 必須: すべての質問はこのフォーマットを使用すること

### ルール: チャットで直接質問を行ってはならない
**重要**: 質問をチャット内で直接行ってはいけません。すべての質問はファイルに配置する必要があります（専用の質問ファイル、または計画ファイル等への埋め込み形式）。

### ルール: 本規約は埋め込み質問にも適用される
**重要**: 本ガイドで定める質問フォーマット規約（最後の選択肢としての「X) その他」必須化、選択肢構造、[Answer]: タグのリテラル出力など）は、専用質問ファイル (`{フェーズ名}-questions.md`) に限らず、計画ファイル (`aidlc-docs/construction/plans/{unit-name}-*-plan.md` や `aidlc-docs/inception/plans/*.md` 等) に**埋め込まれる質問**にも例外なく適用される。ファイル種別を理由に規約をスキップしてはならない。

### ルール: [Answer]: タグは装飾なしのリテラルで出力する
**重要**: 生成ファイル（質問ファイル、計画ファイルなど）に書き出す [Answer]: タグは、行頭から**素のテキスト**として出力しなければならない。バッククォート（例: `` `[Answer]:` ``）、太字（例: `**[Answer]:**`）、斜体、見出し、引用、リスト項目接頭辞など、いかなる Markdown 修飾も付与してはならない。出力は常に以下の形で行頭から始まる:

```markdown
[Answer]:
```

### 質問ファイルのフォーマット

#### ファイル命名規則
- わかりやすい名前を使用する: `{フェーズ名}-questions.md`
- 例:
  - `classification-questions.md`
  - `requirements-questions.md`
  - `story-planning-questions.md`
  - `design-questions.md`

#### 質問の構造
すべての質問には意味のある選択肢と、最後のオプションとして「その他」を含める必要があります:

```markdown
## Question [番号]
[明確で具体的な質問文]

A) [1 番目の意味のある選択肢]
B) [2 番目の意味のある選択肢]
[...必要に応じて追加の選択肢...]
X) その他（下記の [Answer]: タグの後に記述してください）

[Answer]:
```

**重要**:
- 「Other (その他)」はすべての質問の**最後の**オプションとして必須
- 意味のある選択肢のみを含める — 選択肢を埋めるために作り話をしない
- 意味上必要な数または少数の選択肢を使用する（最低 2 つ + その他）

### 完全な例

```markdown
# 要件確認質問

以下の質問に回答して要件の明確化にご協力ください。

## Question 1
主なユーザー認証方式は何ですか？

A) ユーザー名とパスワード
B) ソーシャルメディアログイン（Google、Facebook）
C) シングルサインオン（SSO）
D) 多要素認証
E) その他（下記の [Answer]: タグの後に記述してください）

[Answer]:

## Question 2
これはウェブアプリケーションかモバイルアプリケーションですか？

A) ウェブアプリケーション
B) モバイルアプリケーション
C) ウェブとモバイルの両方
D) その他（下記の [Answer]: タグの後に記述してください）

[Answer]:

## Question 3
これは新規プロジェクトですか、それとも既存のコードベースですか？

A) 新規プロジェクト（グリーンフィールド）
B) 既存のコードベース（ブラウンフィールド）
C) その他（下記の [Answer]: タグの後に記述してください）

[Answer]:
```

### ユーザー回答のフォーマット
ユーザーは [Answer]: タグの後に文字の選択肢を記入して回答します:

```markdown
## Question 1
主なユーザー認証方式は何ですか？

A) ユーザー名とパスワード
B) ソーシャルメディアログイン（Google、Facebook）
C) シングルサインオン（SSO）
D) 多要素認証

[Answer]: C
```

### ユーザー回答の読み取り
ユーザーが完了を確認した後:
1. 質問ファイルを読み込む
2. 回答を [Answer]: タグの後から抽出する
3. すべての質問が回答されていることを検証する
4. 回答に基づいて分析を進める

### 多肢選択問題のガイドライン

#### 選択肢の数
- 最小: 意味のある選択肢 2 つ + 「その他」（A, B, C）
- 通常: 意味のある選択肢 3〜4 つ + 「その他」（A, B, C, D, E）
- 最大: 意味のある選択肢 5 つ + 「その他」（A, B, C, D, E, F）
- **重要**: 選択肢を埋めるために作り話をしない — 意味のある選択肢のみを含める

#### 選択肢の品質
- 選択肢を互いに排他的にする
- 最も一般的なシナリオをカバーする
- 意味のある現実的な選択肢のみを含める
- **「その他」を常に最後のオプションとして含める**（必須）
- 具体的かつ明確にする
- **A、B、C、D の枠を埋めるための選択肢を作り話しない**

#### 良い例:
```markdown
## Question 5
使用するデータベース技術は何ですか？

A) リレーショナル（PostgreSQL、MySQL）
B) NoSQL ドキュメント（MongoDB、DynamoDB）
C) NoSQL キーバリュー（Redis、Memcached）
D) グラフデータベース（Neo4j、Neptune）
E) その他（下記の [Answer]: タグの後に記述してください）

[Answer]:
```

#### 悪い例（避けること）:
```markdown
## Question 5
どのデータベースを使用しますか？

A) はい
B) いいえ
C) わからない

[Answer]:
```

### ワークフローへの統合

#### ステップ 1: 質問ファイルを作成する
```markdown
aidlc-docs/{フェーズ名}-questions.md をすべての質問で作成する
```

#### ステップ 2: ユーザーに通知する
```
「{フェーズ名}-questions.md を [X] 件の質問で作成しました。
[Answer]: タグの後に文字の選択肢を記入して各質問に回答してください。
選択肢がニーズに合わない場合は、最後のオプション（その他）を選んで回答を記述してください。完了したらお知らせください。」
```

#### ステップ 3: 確認を待つ
ユーザーが「完了」「終わった」「できた」などと言うまで待機する。

#### ステップ 4: 読み取りと分析
```
aidlc-docs/{フェーズ名}-questions.md を読み込む
すべての回答を抽出する
完全性を検証する
分析を進める
```

### エラー処理

#### 回答が欠けている場合
空の [Answer]: タグを検出した場合:
```
「Question [X] が回答されていないことに気付きました。先に進む前に、すべての質問に文字の選択肢で回答してください。」
```

#### 無効な回答
回答が有効な文字の選択肢でない場合:
```
「Question [X] の回答「[回答]」は無効です。
質問に記載された文字の選択肢のみを使用してください。」
```

#### 曖昧な回答
ユーザーが文字ではなく説明を提供した場合:
```
「Question [X] では、回答に最も近い文字の選択肢を選んでください。
該当する選択肢がない場合は、「その他」を選んで [Answer]: タグの後に説明を追加してください。」
```

### 矛盾と曖昧さの検出

**必須**: ユーザーの回答を読み取った後、必ず矛盾と曖昧さを確認すること。

#### 矛盾の検出
論理的に矛盾する回答を探す:
- スコープの不一致: 「バグ修正」だが「コードベース全体に影響」
- リスクの不一致: 「低リスク」だが「破壊的な変更」
- タイムラインの不一致: 「クイックフィックス」だが「複数のサブシステム」
- 影響の不一致: 「単一コンポーネント」だが「アーキテクチャの大幅な変更」

#### 曖昧さの検出
不明確または境界線上の回答を探す:
- 複数の分類に当てはまる可能性がある回答
- 具体性に欠ける回答
- 複数の質問間で矛盾する指標

#### 明確化質問の作成
矛盾または曖昧さが検出された場合:

1. **明確化ファイルを作成する**: `{フェーズ名}-clarification-questions.md`
2. **問題を説明する**: 何の矛盾/曖昧さが検出されたかを明確に述べる
3. **的を絞った質問をする**: 多肢選択形式で問題を解消する
4. **元の質問を参照する**: どの質問で矛盾する回答があったかを示す

**例**:
```markdown
# [フェーズ名] 明確化質問

回答に矛盾が見つかり、明確化が必要です:

## 矛盾 1: [簡単な説明]
あなたは「[回答 A]」（Q[X]:[文字]）と答えましたが、同時に「[回答 B]」（Q[Y]:[文字]）とも答えています。
これらの回答は[説明]のため矛盾しています。

### 明確化質問 1
[矛盾を解消するための具体的な質問]

A) [最初の回答に向かう解決策]
B) [2 番目の回答に向かう解決策]
C) [中間の選択肢]
D) [質問を再定義する選択肢]

[Answer]:

## 曖昧さ 1: [簡単な説明]
Q[X] への回答（「[回答]」）は[説明]のため曖昧です。

### 明確化質問 2
[曖昧さを解消するための具体的な質問]

A) [明確な選択肢 1]
B) [明確な選択肢 2]
C) [明確な選択肢 3]
D) [明確な選択肢 4]

[Answer]:
```

#### 明確化のワークフロー

1. **検出**: すべての回答を矛盾/曖昧さについて分析する
2. **作成**: 問題が見つかった場合は明確化質問ファイルを生成する
3. **通知**: 問題と明確化ファイルについてユーザーに伝える
4. **待機**: ユーザーが明確化を提供するまで進行しない
5. **再検証**: 明確化の後、再度一貫性を確認する
6. **進行**: すべての矛盾が解消された場合のみ先に進む

#### ユーザーへのメッセージ例
```
「あなたの回答に 2 つの矛盾が見つかりました:

1. バグ修正のスコープとコードベースへの影響（Q1 vs Q2）
2. 低リスクと破壊的な変更（Q7 vs Q4）

これらを解消するために classification-clarification-questions.md に 2 つの質問を作成しました。
処理を進める前に、これらの確認の質問に回答してください。」
```

### ベストプラクティス

1. **具体的に**: 質問は明確で曖昧さがないようにする
2. **包括的に**: 必要なすべての情報をカバーする
3. **簡潔に**: 質問は 1 つのトピックに集中させる
4. **実用的に**: 選択肢は現実的で実行可能にする
5. **一貫性を保つ**: すべての質問ファイルで同じフォーマットを使用する

## まとめ

**留意事項**:
- 常に質問ファイルを作成する
- 常に多肢選択形式を使用する
- **常に「Other (その他)」を最後のオプションとして含める（必須）**
- 意味のある選択肢のみを含める — 選択肢を埋めるために作り話をしない
- 常に [Answer]: タグを使用する
- 常にユーザーの完了を待つ
- 常に回答の矛盾を検証する
- 必要に応じて常に明確化ファイルを作成する
- 常に進行前に矛盾を解消する
- チャットで質問しない
- A、B、C、D の枠を埋めるための選択肢を作り話しない
- 回答なしで進行しない
- 未解消の矛盾がある状態で進行しない
- 曖昧な回答に基づいて仮定しない
