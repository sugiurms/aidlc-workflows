# コード生成 — 詳細手順

## 概要
このステージでは、2 つの統合されたパートを通じて各作業単位のコードを生成します:
- **パート 1 — 計画**: 明示的なステップを含む詳細なコード生成計画を作成する
- **パート 2 — 生成**: 承認された計画を実行してコード、テスト、アーティファクトを生成する

**注記**: ブラウンフィールドプロジェクトの場合、「生成」とは重複コピーを作成するのではなく、適切な場合に既存のファイルを修正することを意味します。

## 前提条件
- ユニットのユニット設計生成が完了していること
- 非機能要件実装（実行した場合）がユニットに対して完了していること
- すべてのユニット設計アーティファクトが利用可能であること
- ユニットがコード生成の準備ができていること

# パート 1: 計画

## ステップ 1: ユニットコンテキストの分析
- [ ] ユニット設計生成からユニット設計アーティファクトを読み込む
- [ ] ユニットストーリーマップを読み込んで割り当てられたストーリーを理解する
- [ ] ユニットの依存関係とインターフェースを特定する
- [ ] ユニットがコード生成の準備ができていることを検証する

## ステップ 2: 詳細なユニットコード生成計画の作成
- [ ] `aidlc-docs/aidlc-state.md` からプロジェクトタイプを読み込む（ワークスペースルートは git リポジトリルート（`git rev-parse --show-toplevel`）として実行時に解決する）
- [ ] コードの配置先を決定する（構造パターンについては重要なルールを参照）
- [ ] **ブラウンフィールドのみ**: リバースエンジニアリングの code-structure.md を確認して修正する既存ファイルを見つける
- [ ] 正確なパスを記録する（aidlc-docs/ には絶対に配置しない）
- [ ] ユニット生成の明示的な手順を作成する:
  - プロジェクト構造のセットアップ（グリーンフィールドのみ）
  - ビジネスロジックの生成
  - ビジネスロジックのユニットテスト
  - ビジネスロジックのサマリー
  - API 層の生成
  - API 層のユニットテスト
  - API 層のサマリー
  - リポジトリ層の生成
  - リポジトリ層のユニットテスト
  - リポジトリ層のサマリー
  - フロントエンドコンポーネントの生成（該当する場合）
  - フロントエンドコンポーネントのユニットテスト（該当する場合）
  - フロントエンドコンポーネントのサマリー（該当する場合）
  - データベースマイグレーションスクリプト（データモデルが存在する場合）
  - ドキュメントの生成（API ドキュメント、README の更新）
  - デプロイメントアーティファクトの生成
- [ ] 各ステップに連番を付ける
- [ ] ストーリーマッピングの参照を含める
- [ ] 各ステップにチェックボックス [ ] を追加する

## ステップ 3: ユニット生成コンテキストの含め方
- [ ] このユニットに対して以下を含める:
  - このユニットで実装されるストーリー
  - 他のユニット/サービスへの依存関係
  - 期待されるインターフェースとコントラクト
  - このユニットが所有するデータベースエンティティ
  - サービス境界と責務

## ステップ 4: ユニット計画ドキュメントの作成
- [ ] 完全な計画を `aidlc-docs/construction/plans/{unit-name}-code-generation-plan.md` として保存する
- [ ] ステップ番号（ステップ 1、ステップ 2 など）を含める
- [ ] ユニットコンテキストと依存関係を含める
- [ ] ストーリーのトレーサビリティを含める
- [ ] 計画がステップバイステップで実行可能であることを確認する
- [ ] この計画がコード生成の単一の情報源であることを強調する

## ステップ 5: ユニット計画のサマリー提示
- [ ] ユニットコード生成計画のサマリーをユーザーに提供する
- [ ] ユニット生成アプローチを強調する
- [ ] ステップの順序とストーリーのカバレッジを説明する
- [ ] 総ステップ数と推定スコープをメモする

## ステップ 6: 承認プロンプトの記録
- [ ] 承認を求める前に、タイムスタンプ付きで `aidlc-docs/audit.md` にプロンプトを記録する
- [ ] 完全なユニットコード生成計画への参照を含める
- [ ] ISO 8601 タイムスタンプフォーマットを使用する

## ステップ 7: 明示的な承認の待機
- [ ] ユーザーがユニットコード生成計画を明示的に承認するまで進行しない
- [ ] 承認は計画全体と生成順序をカバーする必要がある
- [ ] ユーザーが変更を要求した場合は計画を更新して承認プロセスを繰り返す

## ステップ 8: 承認応答の記録
- [ ] ユーザーの承認応答をタイムスタンプ付きで `aidlc-docs/audit.md` に記録する
- [ ] ユーザーの正確な応答テキストを含める
- [ ] 承認ステータスを明確にマークする

## ステップ 9: 進捗の更新
- [ ] `aidlc-state.md` でコード生成パート 1（計画）を完了としてマークする
- [ ] 「現在のステータス」セクションを更新する
- [ ] コード生成への移行の準備をする

# パート 2: 生成

## ステップ 10: ユニットコード生成計画の読み込み
- [ ] `aidlc-docs/construction/plans/{unit-name}-code-generation-plan.md` から完全な計画を読み込む
- [ ] 次の未完了ステップ（最初の [ ] チェックボックス）を特定する
- [ ] そのステップのコンテキスト（ユニット、依存関係、ストーリー）を読み込む

## ステップ 11: 現在のステップの実行
- [ ] 計画からターゲットディレクトリを確認する（aidlc-docs/ には絶対に配置しない）
- [ ] **ブラウンフィールドのみ**: ターゲットファイルが存在するか確認する
- [ ] 現在のステップが説明した内容を正確に生成する:
  - **ファイルが存在する場合**: その場で修正する（`ClassName_modified.java`、`ClassName_new.java` などを作成しない）
  - **ファイルが存在しない場合**: 新しいファイルを作成する
- [ ] 正しい場所に書き込む:
  - **アプリケーションコード**: プロジェクト構造に従ったワークスペースルート
  - **ドキュメント**: `aidlc-docs/construction/{unit-name}/code/`（Markdown のみ）
  - **ビルド/設定ファイル**: ワークスペースルート
- [ ] ユニットストーリーの要件に従う
- [ ] 依存関係とインターフェースを尊重する

## ステップ 12: 進捗の更新
- [ ] ユニットコード生成計画で完了したステップを [x] としてマークする
- [ ] 生成が完了したときに関連するユニットストーリーを [x] としてマークする
- [ ] `aidlc-docs/aidlc-state.md` の現在のステータスを更新する
- [ ] **ブラウンフィールドのみ**: 重複ファイルが作成されていないことを確認する（例: `ClassName.java` と並んで `ClassName_modified.java` が存在しない）
- [ ] 生成されたすべてのアーティファクトを保存する

## ステップ 13: 生成の継続または完了
- [ ] まだステップが残っている場合はステップ 10 に戻る
- [ ] すべてのステップが完了した場合は完了メッセージの提示に進む

## ステップ 14: 完了メッセージの提示
以下の構造で完了メッセージを提示する:
     1. **完了告知**（必須）: 常にこれで始める:

```markdown
# 💻 Code Generation Complete - [unit-name]
```

     2. **AI サマリー**（省略可）: 構造化された箇条書きサマリーを提供する
        - **ブラウンフィールド**: 修正したファイルと作成したファイルを区別する（例: 「• Modified: `src/services/user-service.ts`」、「• Created: `src/services/auth-service.ts`」）
        - **グリーンフィールド**: 作成したファイルをパス付きでリストする（例:「• Created: `src/services/user-service.ts`」）
        - テスト、ドキュメント、デプロイメントアーティファクトをパス付きでリストする
        - 事実のみを記載し、ワークフローの指示を含めない
     3. **フォーマット済みワークフローメッセージ**（必須）: 常にこの正確なフォーマットで終わる:

```markdown
> **📋 レビューをお願いします:**
> 以下の生成されたコードを確認してください:
> - **アプリケーションコード**: `[actual-workspace-path]`
> - **ドキュメント**: `aidlc-docs/construction/[unit-name]/code/`



> **🚀 次のステップ**
>
> **以下から選択してください:**
>
> 🔧 **変更を依頼** - レビュー内容に基づいて生成されたコードの変更を依頼する
> ✅ **次のステージへ進む** - コード生成を承認して **[次のユニット/ビルドとテスト]** に進む

---
```

## ステップ 15: 明示的な承認の待機
- ユーザーが生成されたコードを明示的に承認するまで進行しない
- 承認は明確で曖昧さのないものでなければならない
- ユーザーが変更を要求した場合はコードを更新して承認プロセスを繰り返す

## ステップ 16: 承認の記録と進捗の更新
- タイムスタンプ付きで audit.md に承認を記録する
- タイムスタンプ付きでユーザーの承認応答を記録する
- aidlc-state.md でこのユニットのコード生成ステージを完了としてマークする

## 重要なルール

### コードの配置ルール
- **ワークスペースルート**: git リポジトリルート（`git rev-parse --show-toplevel` で取得）として実行時に解決する（クローン先に依存する絶対パスはどのファイルにも記録しない）
- **アプリケーションコード**: ワークスペースルートのみ（aidlc-docs/ には絶対に配置しない）
- **ドキュメント**: aidlc-docs/ のみ（Markdown サマリー）

**プロジェクトタイプ別の構造パターン**:
- **ブラウンフィールド**: 既存の構造を使用する（例: `src/main/java/`、`lib/`、`pkg/`）
- **グリーンフィールド・単一ユニット**: ワークスペースルートに `src/`、`tests/`、`config/`
- **グリーンフィールド・マルチユニット（マイクロサービス）**: `{unit-name}/src/`、`{unit-name}/tests/`
- **グリーンフィールド・マルチユニット（モノリス）**: `src/{unit-name}/`、`tests/{unit-name}/`

### ブラウンフィールドファイル修正のルール
- 生成前にファイルが存在するか確認する
- 存在する場合: その場で修正する（`ClassName_modified.java` などのコピーを作成しない）
- 存在しない場合: 新しいファイルを作成する
- 生成後に重複ファイルがないことを確認する（ステップ 12）

### 計画フェーズのルール
- すべての生成活動に対して明示的な番号付きステップを作成する
- 計画にストーリーのトレーサビリティを含める
- ユニットのコンテキストと依存関係を記録する
- 生成の前に明示的なユーザー承認を得る

### 生成フェーズのルール
- **ハードコードされたロジック禁止**: ユニット計画に書かれていることのみ実行する
- **計画に厳密に従う**: ステップの順序から逸脱しない
- **チェックボックスを更新する**: 各ステップ完了直後に [x] をマークする
- **ストーリーのトレーサビリティ**: 機能が実装されたらユニットストーリーを [x] としてマークする
- **依存関係を尊重する**: ユニットの依存関係が満たされている場合のみ実装する

### 自動化フレンドリーなコードのルール
UI コード（ウェブ、モバイル、デスクトップ）を生成する場合、要素が自動化フレンドリーであることを確認する:
- インタラクティブな要素（ボタン、入力フィールド、リンク、フォーム）に `data-testid` 属性を追加する
- 一貫した命名を使用する: `{component}-{element-role}`（例: `login-form-submit-button`、`user-list-search-input`）
- レンダリング間で変化する動的または自動生成の ID を避ける
- `data-testid` の値はコード変更が行われても安定させる（要素の目的が変わった場合のみ変更する）

## 完了基準
- 完全なユニットコード生成計画が作成され承認されていること
- ユニットコード生成計画内のすべてのステップが [x] でマークされていること
- 計画に従ってすべてのユニットストーリーが実装されていること
- すべてのコードとテストが生成されていること（テストはビルドとテストフェーズで実行される）
- デプロイメントアーティファクトが生成されていること
- ビルドと検証の準備ができた完全なユニットが揃っていること
