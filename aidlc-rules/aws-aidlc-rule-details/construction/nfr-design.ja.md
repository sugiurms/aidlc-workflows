# 非機能要件設計

## 前提条件
- ユニットの非機能要件が完了していること
- 非機能要件アーティファクトが利用可能であること
- 実行計画が非機能要件設計ステージを実行すべきことを示していること

## 概要
パターンと論理コンポーネントを使用して非機能要件をユニット設計に組み込む。

## 実行手順

### ステップ 1: 非機能要件の分析
- `aidlc-docs/construction/{unit-name}/nfr-requirements/` から非機能要件を読み込む
- スケーラビリティ、パフォーマンス、可用性、セキュリティのニーズを理解する

### ステップ 2: 非機能要件設計計画の作成
- 非機能要件設計のためのチェックボックス [] を含む計画を生成する
- 設計パターンと論理コンポーネントに集中する
- 各ステップにチェックボックス [] を持たせる

### ステップ 3: コンテキストに適した質問の生成
**ディレクティブ**: 非機能要件を徹底的に分析して、確認によって非機能要件設計の品質が向上するすべての領域を特定する。包括的な非機能設計カバレッジを確保するために積極的に質問する。

**重要**: 非機能要件設計の品質に影響する可能性のある曖昧さや詳細の欠如がある場合は、デフォルトで質問する。非機能要件パターンについて誤った想定をするよりも質問が多すぎる方が良い。

**必須**: 以下のすべてのカテゴリを評価し、各カテゴリに対して的を絞った質問をする。各カテゴリの適用可能性は非機能要件から判断する — 明確な理由なくカテゴリをスキップしない:

- `[Answer]:` タグフォーマットを使用して質問を埋め込む
- 曖昧さ、欠けている情報、または明確化が必要な領域に集中する
- ユーザー入力がパターンとコンポーネントの決定を改善するすべての箇所に質問を生成する
- **疑問がある場合は質問する** — 過信は非機能設計の品質を下げる

**評価すべき質問カテゴリ**（すべてのカテゴリを考慮すること）:
- **耐障害性パターン** — フォールトトレランスのアプローチ、リトライ戦略、障害回復の期待について質問する
- **スケーラビリティパターン** — スケーリングメカニズム、負荷境界、成長予測について質問する
- **パフォーマンスパターン** — 最適化戦略、レイテンシ目標、スループット要件について質問する
- **セキュリティパターン** — セキュリティ実装アプローチ、脅威モデル、コンプライアンス制約について質問する
- **論理コンポーネント** — インフラストラクチャコンポーネント（キュー、キャッシュ、サーキットブレーカーなど）とその統合パターンについて質問する

### ステップ 4: 計画の保存
- `aidlc-docs/construction/plans/{unit-name}-nfr-design-plan.md` として保存する
- ユーザー入力のためのすべての `[Answer]:` タグを含める

### ステップ 5: 回答の収集と分析
- ユーザーがすべての `[Answer]:` タグを完了するまで待機する
- 曖昧または不明確な応答を確認する
- 必要に応じてフォローアップ質問を追加する

### ステップ 6: 非機能要件設計アーティファクトの生成
- `aidlc-docs/construction/{unit-name}/nfr-design/nfr-design-patterns.md` を作成する
- `aidlc-docs/construction/{unit-name}/nfr-design/logical-components.md` を作成する

### ステップ 7: 完了メッセージの提示
以下の構造で完了メッセージを提示する:
     1. **完了告知**（必須）: 常にこれで始める:

```markdown
# 🎨 NFR Design Complete - [unit-name]
```

     2. **AI サマリー**（省略可）: 非機能要件設計の構造化された箇条書きサマリーを提供する
        - フォーマット: 「非機能要件設計によって [説明] が組み込まれました:」
        - 実装された主要な設計パターンをリストする（箇条書き）
        - 論理コンポーネントとインフラストラクチャ要素をリストする
        - 適用された耐障害性、スケーラビリティ、パフォーマンスパターンについて言及する
        - ワークフローの指示を含めない（「レビューしてください」「お知らせください」「次のフェーズに進みます」「進める前に」）
        - 事実のみで内容に集中する
     3. **フォーマット済みワークフローメッセージ**（必須）: 常にこの正確なフォーマットで終わる:

```markdown
> **📋 <u>**レビューをお願いします:**</u>**
> 以下の非機能要件設計を確認してください: `aidlc-docs/construction/[unit-name]/nfr-design/`



> **🚀 <u>**次のステップ**</u>**
>
> **以下から選択してください:**
>
> 🔧 **変更を依頼** - レビュー内容に基づいて非機能要件設計の変更を依頼する
> ✅ **次のステージへ進む** - 非機能要件設計を承認して **[次のステージ名]** に進む

---
```

### ステップ 8: 明示的な承認の待機
- ユーザーが非機能要件設計を明示的に承認するまで進行しない
- 承認は明確で曖昧さのないものでなければならない
- ユーザーが変更を要求した場合は設計を更新して承認プロセスを繰り返す

### ステップ 9: 承認の記録と進捗の更新
- タイムスタンプ付きで audit.md に承認を記録する
- タイムスタンプ付きでユーザーの承認応答を記録する
- aidlc-state.md で非機能要件設計ステージを完了としてマークする
