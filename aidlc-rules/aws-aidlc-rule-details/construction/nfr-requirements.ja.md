# 非機能要件

## 前提条件
- ユニットの機能設計が完了していること
- ユニットの機能設計アーティファクトが利用可能であること
- 実行計画が非機能要件ステージを実行すべきことを示していること

## 概要
ユニットの非機能要件を決定し、技術スタックを選択する。

## 実行手順

### ステップ 1: 機能設計の分析
- `aidlc-docs/construction/{unit-name}/functional-design/` から機能設計アーティファクトを読み込む
- ビジネスロジックの複雑さと要件を理解する

### ステップ 2: 非機能要件計画の作成
- 非機能要件評価のためのチェックボックス [] を含む計画を生成する
- スケーラビリティ、パフォーマンス、可用性、セキュリティに集中する
- 各ステップにチェックボックス [] を持たせる

### ステップ 3: コンテキストに適した質問の生成
**ディレクティブ**: 機能設計を徹底的に分析して、非機能要件の確認によってシステム品質とアーキテクチャの決定が向上するすべての領域を特定する。包括的な非機能要件カバレッジを確保するために積極的に質問する。

**重要**: システム品質に影響する可能性のある曖昧さや詳細の欠如がある場合は、デフォルトで質問する。誤った非機能要件の想定よりも質問が多すぎる方が良い。

- `[Answer]:` タグフォーマットを使用して質問を埋め込む
- 曖昧さ、欠けている情報、または明確化が必要な領域に集中する
- ユーザー入力が非機能要件と技術スタックの決定を改善するすべての箇所に質問を生成する
- **疑問がある場合は質問する** — 過信はシステム品質を下げる

**評価すべき質問カテゴリ**（すべてのカテゴリを考慮すること）:
- **スケーラビリティ要件** — 想定負荷、成長パターン、スケーリングトリガー、キャパシティプランニングについて質問する
- **パフォーマンス要件** — レスポンスタイム、スループット、レイテンシ、パフォーマンスベンチマークについて質問する
- **可用性要件** — アップタイムの期待値、障害復旧、フェイルオーバー、事業継続性について質問する
- **セキュリティ要件** — データ保護、コンプライアンス、認証、認可、脅威モデルについて質問する
- **技術スタック選択** — テクノロジーの好み、制約、既存システム、統合要件について質問する
- **信頼性要件** — エラー処理、フォールトトレランス、モニタリング、アラートのニーズについて質問する
- **保守性要件** — コード品質、ドキュメント、テスト、運用要件について質問する
- **ユーザビリティ要件** — ユーザーエクスペリエンス、アクセシビリティ、インターフェース要件について質問する

### ステップ 4: 計画の保存
- `aidlc-docs/construction/plans/{unit-name}-nfr-requirements-plan.md` として保存する
- ユーザー入力のためのすべての `[Answer]:` タグを含める

### ステップ 5: 回答の収集と分析
- ユーザーがすべての `[Answer]:` タグを完了するまで待機する
- **必須**: 曖昧または不明確な回答についてすべての応答を慎重に確認する
- **重要**: 不明確な応答にはフォローアップ質問を追加する — 曖昧さがある状態で進行しない
- 「依存する」「多分」「わからない」「混在」「どちらかというと」「標準的な」「典型的な」などの応答を探す
- 曖昧さが検出された場合は明確化質問ファイルを作成する
- **すべての曖昧さが解消されるまで進行しない**

### ステップ 6: 非機能要件アーティファクトの生成
- `aidlc-docs/construction/{unit-name}/nfr-requirements/nfr-requirements.md` を作成する
- `aidlc-docs/construction/{unit-name}/nfr-requirements/tech-stack-decisions.md` を作成する

### ステップ 7: 完了メッセージの提示
以下の構造で完了メッセージを提示する:
     1. **完了告知**（必須）: 常にこれで始める:

```markdown
# 📊 NFR Requirements Complete - [unit-name]
```

     2. **AI サマリー**（省略可）: 非機能要件の構造化された箇条書きサマリーを提供する
        - フォーマット: 「非機能要件評価によって [説明] が特定されました:」
        - 主要なスケーラビリティ、パフォーマンス、可用性要件をリストする（箇条書き）
        - 特定されたセキュリティとコンプライアンス要件をリストする
        - 技術スタックの決定とその根拠について言及する
        - ワークフローの指示を含めない（「レビューしてください」「お知らせください」「次のフェーズに進みます」「進める前に」）
        - 事実のみで内容に集中する
     3. **フォーマット済みワークフローメッセージ**（必須）: 常にこの正確なフォーマットで終わる:

```markdown
> **📋 <u>**REVIEW REQUIRED:**</u>**  
> Please examine the NFR requirements at: `aidlc-docs/construction/[unit-name]/nfr-requirements/`



> **🚀 <u>**WHAT'S NEXT?**</u>**
>
> **You may:**
>
> 🔧 **Request Changes** - Ask for modifications to the NFR requirements based on your review  
> ✅ **Continue to Next Stage** - Approve NFR requirements and proceed to **[next-stage-name]**

---
```

### ステップ 8: 明示的な承認の待機
- ユーザーが非機能要件を明示的に承認するまで進行しない
- 承認は明確で曖昧さのないものでなければならない
- ユーザーが変更を要求した場合は要件を更新して承認プロセスを繰り返す

### ステップ 9: 承認の記録と進捗の更新
- タイムスタンプ付きで audit.md に承認を記録する
- タイムスタンプ付きでユーザーの承認応答を記録する
- aidlc-state.md で非機能要件ステージを完了としてマークする
