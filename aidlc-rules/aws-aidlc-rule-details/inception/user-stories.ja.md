# ユーザーストーリー — 詳細な手順

## 目的
**要件をユーザー中心のストーリーと受け入れ基準に変換する**

ユーザーストーリーが対象とする内容:
- ビジネス要件をユーザー中心のナラティブに変換する
- 各ストーリーに対して明確な受け入れ基準を定義する
- 異なるステークホルダータイプを表すユーザーペルソナを作成する
- チーム全体での共通理解を確立する
- 実装のためのテスト可能な仕様を提供する

## 前提条件
- ワークスペース検出が完了していること
- 要件分析を推奨（利用可能な場合は要件を参照できる）
- ワークフロー計画がユーザーストーリーステージを実行すべきことを示していること

## インテリジェントな評価ガイドライン

**ユーザーストーリーを実行するタイミング**: 進行する前にこの強化された評価を使用する:

### 高優先度の実行（常に実行）
- **新しいユーザー機能**: ユーザーが直接インタラクションする新しい機能
- **ユーザーエクスペリエンスの変更**: 既存のユーザーワークフローまたはインターフェースへの変更
- **マルチペルソナシステム**: 異なるタイプのユーザーを対象とするアプリケーション
- **顧客向け API**: 外部ユーザーまたはシステムが使用するサービス
- **複雑なビジネスロジック**: 複数のシナリオまたはビジネスルールを持つ要件
- **クロスチームプロジェクト**: 複数のチームにまたがる共通理解が必要な作業

### 中優先度の実行（複雑さを評価）
- **バックエンドのユーザー影響**: ユーザーエクスペリエンスに間接的に影響する内部変更
- **パフォーマンスの改善**: ユーザーに見えるメリットがある強化
- **統合作業**: ユーザーワークフローに影響するシステムの接続
- **データの変更**: ユーザーデータ、レポート、または分析に影響する変更
- **セキュリティ強化**: ユーザー認証または権限に影響する変更

### 複雑さの評価要素
中優先度のケースでは、以下のいずれかに該当する場合にユーザーストーリーを実行する:
- **スコープ**: 変更が複数のコンポーネントまたはユーザータッチポイントにまたがる
- **曖昧さ**: 要件にストーリーが明確にできる不明確な側面がある
- **リスク**: 高いビジネスインパクトまたは誤解の可能性がある
- **ステークホルダー**: 複数のビジネスステークホルダーが要件に関与している
- **テスト**: ユーザー受け入れテストが必要になる
- **オプション**: 複数の有効な実装アプローチが存在する

### シンプルなケースのみスキップ
- **純粋なリファクタリング**: ユーザーへの影響がゼロの内部コード改善
- **孤立したバグ修正**: 明確なスコープを持つシンプルで明確に定義された修正
- **インフラストラクチャのみ**: ユーザー向けの影響がない変更
- **開発者ツール**: ビルドプロセス、CI/CD、または開発環境の変更
- **ドキュメント**: 機能に影響しない更新

### デフォルトの決定ルール
**疑問がある場合は、ユーザーストーリーを含め、明確化のための質問をする。** 適切な明確化を含む包括的なストーリーを作成するオーバーヘッドは、通常、以下のメリットによって上回られる:
- より明確な要件の理解
- より良いチームの整合性
- 改善されたテスト基準
- ステークホルダーとのコミュニケーションの向上
- 実装リスクの軽減
- 開発中のコストのかかる変更の減少
- より良いユーザーエクスペリエンスの結果

# パート 1: 計画

## ステップ 1: ユーザーストーリーの必要性の検証（必須）

**重要**: ユーザーストーリーを進める前に、この評価を実行する:

### 評価プロセス
1. **リクエストコンテキストの分析**:
   - 元のユーザーリクエストと要件を確認する
   - ユーザー向けの変更と内部のみの変更を識別する
   - 作業の複雑さとスコープを評価する
   - ビジネスステークホルダーの関与を評価する

2. **評価基準の適用**:
   - 高優先度の指標をチェックする（常に実行）
   - 中優先度の要素を評価する（複雑さベースの決定）
   - これがスキップすべきシンプルなケースでないことを確認する

3. **評価決定の文書化**:
   - `aidlc-docs/inception/plans/user-stories-assessment.md` を作成する
   - このリクエストにユーザーストーリーが価値ある理由を含める
   - 適用される具体的な評価基準を参照する
   - 期待されるメリット（明確さ、テスト、ステークホルダーの整合性）を説明する

4. **正当化された場合のみ進行**:
   - ユーザーストーリーはプロジェクトに明確な価値を追加しなければならない
   - 評価は具体的なメリットがオーバーヘッドを上回ることを示さなければならない
   - 決定はプロジェクトのステークホルダーに対して説明可能でなければならない

### 評価ドキュメントテンプレート
```markdown
# User Stories Assessment

## Request Analysis
- **Original Request**: [簡潔なサマリー]
- **User Impact**: [Direct/Indirect/None]
- **Complexity Level**: [Simple/Medium/Complex]
- **Stakeholders**: [関与する当事者のリスト]

## Assessment Criteria Met
- [ ] High Priority: [適用される基準のリスト]
- [ ] Medium Priority: [複雑さの正当化を含む適用される基準のリスト]
- [ ] Benefits: [ユーザーストーリーから期待される価値]

## Decision
**Execute User Stories**: [Yes/No]
**Reasoning**: [詳細な正当化]

## Expected Outcomes
- [ユーザーストーリーが提供する具体的なメリットのリスト]
- [ストーリーがプロジェクトの成功をどのように改善するか]
```

## ステップ 2: ストーリー計画の作成
- プロダクトオーナーの役割を担う
- ストーリー開発のためのステップバイステップの実行チェックリストを含む包括的な計画を生成する
- 各ステップとサブステップにチェックボックス [] を持たせる
- 要件をユーザーストーリーに変換するための方法論とアプローチに集中する

## ステップ 3: コンテキストに適した質問の生成
**ディレクティブ**: 要件とコンテキストを徹底的に分析して、確認によってストーリーの品質とチームの理解が向上するすべての領域を特定する。包括的なユーザーストーリー開発を確保するために積極的に質問する。

**重要**: ストーリーの品質に影響する可能性のある曖昧さや詳細の欠如がある場合は、デフォルトで質問する。不完全または不明確なストーリーを作成するよりも質問が多すぎる方が良い。

**質問のフォーマットルールについては `common/question-format-guide.md` を参照**

- `[Answer]:` タグフォーマットを使用して質問を埋め込む
- 曖昧さ、欠けている情報、または明確化が必要な領域に集中する
- ユーザー入力がストーリー作成の決定を改善するすべての箇所に質問を生成する
- **疑問がある場合は質問する** — 過信は質の悪いストーリーにつながる

**評価すべき質問カテゴリ**（すべてのカテゴリを考慮すること）:
- **ユーザーペルソナ** — ユーザータイプ、役割、特性、動機について質問する
- **ストーリーの粒度** — 適切な詳細レベル、ストーリーサイズ、分解アプローチについて質問する
- **ストーリーフォーマット** — フォーマットの好み、テンプレートの使用、ドキュメント標準について質問する
- **分解アプローチ** — 組織化方法、優先順位付け、グルーピング戦略について質問する
- **受け入れ基準** — 詳細レベル、フォーマット、テストアプローチ、検証方法について質問する
- **ユーザージャーニー** — ユーザーワークフロー、インタラクションパターン、エクスペリエンスフローについて質問する
- **ビジネスコンテキスト** — ビジネス目標、成功指標、ステークホルダーのニーズについて質問する
- **技術的な制約** — 技術的な制限、統合要件、システム境界について質問する

## ステップ 4: 必須ストーリーアーティファクトを計画に含める
- 以下の必須アーティファクトをストーリー計画に**常に**含める:
  - [ ] INVEST 基準に従ったユーザーストーリーを含む stories.md を生成する
  - [ ] ユーザーアーキタイプと特性を含む personas.md を生成する
  - [ ] ストーリーが独立している（Independent）、交渉可能（Negotiable）、価値がある（Valuable）、見積もり可能（Estimable）、小さい（Small）、テスト可能（Testable）であることを確認する
  - [ ] 各ストーリーに受け入れ基準を含める
  - [ ] ペルソナを関連するユーザーストーリーにマッピングする

## ステップ 5: ストーリーオプションの提示
- 計画ドキュメントにストーリー分解の異なるアプローチを含める:
  - **ユーザージャーニーベース**: ストーリーがユーザーワークフローとインタラクションに従う
  - **フィーチャーベース**: ストーリーがシステム機能と能力を中心に組織化される
  - **ペルソナベース**: ストーリーが異なるユーザータイプとそのニーズでグループ化される
  - **ドメインベース**: ストーリーがビジネスドメインまたはコンテキストを中心に組織化される
  - **エピックベース**: ストーリーがサブストーリーを持つ階層的なエピックとして構造化される
- 各アプローチのトレードオフとメリットを説明する
- 明確な決定基準を持つハイブリッドアプローチを許可する

## ステップ 6: ストーリー計画の保存
- 埋め込まれた質問を含む完全なストーリー計画を `aidlc-docs/inception/plans/` ディレクトリに保存する
- ファイル名: `story-generation-plan.md`
- ユーザー入力のためのすべての `[Answer]:` タグを含める
- 計画が包括的ですべてのストーリー開発側面をカバーすることを確認する

## ステップ 7: ユーザー入力の要求
- ユーザーがストーリー計画ドキュメントのすべての `[Answer]:` タグを直接記入するよう求める
- 監査証跡と決定の文書化の重要性を強調する
- `[Answer]:` タグの記入方法に関する明確な指示を提供する
- 進行する前にすべての質問に回答しなければならないことを説明する

## ステップ 8: 回答の収集
- ユーザーがドキュメント内の `[Answer]:` タグを使用してすべての質問に回答するまで待機する
- すべての `[Answer]:` タグが完了するまで進行しない
- `[Answer]:` タグが空白になっていないことをドキュメントで確認する

## ステップ 9: 回答の分析（必須）
進行する前に、すべてのユーザーの回答を慎重に確認しなければならない:
- **曖昧または不明確な応答**: 「混在」「どちらかというと」「わからない」「依存する」「多分」「おそらく」
- **未定義の基準または用語**: 明確な定義なしに概念を参照している
- **矛盾した回答**: 互いに矛盾する応答
- **生成の詳細が欠けている**: 実装のための具体的なガイダンスがない回答
- **オプションを組み合わせている回答**: 明確な決定ルールなしに異なるアプローチを統合した応答
- **不完全な説明**: 明示的に定義されていない外部要素を参照する回答
- **想定に基づく応答**: 明示的に述べられていない知識を想定した回答

## ステップ 10: 必須フォローアップ質問
ステップ 9 の分析で曖昧な回答が明らかになった場合、以下を行わなければならない:
- `[Answer]:` タグを使用して別の明確化質問ファイルを作成する
- すべての曖昧さが完全に解消されるまで承認プロセスに進まない
- **重要**: 徹底的に行う — 不明確な応答すべてに対してフォローアップ質問をする
- 必要なフォローアップの例:
  - 「A と B の混在とおっしゃいましたが、A と B をいつ使うかを決める具体的な基準は何ですか?」
  - 「A と B の中間とおっしゃいましたが、正確な中間アプローチを定義していただけますか?」
  - 「不明とおっしゃいましたが、何の追加情報があれば決定できますか?」
  - 「複雑さによって異なるとおっしゃいましたが、複雑さのレベルとしきい値をどう定義しますか?」
  - 「ハイブリッドアプローチを選択されましたが、各方法をいつ使うかの具体的なルールは何ですか?」
  - 「おそらく X とおっしゃいましたが、確実に X になるか確実に X でなくなるのはどんな要素ですか?」
  - 「標準的な慣行とおっしゃいましたが、その標準的な慣行を定義していただけますか?」

## ステップ 11: 実装の詳細を避ける
- 優先順位付けや開発タスクではなく、ストーリー作成の方法論に集中する
- このステージでは技術的な生成について議論しない
- 開発のタイムラインやスプリント計画の作成を避ける
- ストーリーの構造とフォーマットの決定に集中する

## ステップ 12: 承認プロンプトのログ記録
- 承認を求める前に、タイムスタンプ付きで `aidlc-docs/audit.md` にプロンプトをログに記録する
- 完全な承認プロンプトテキストを含める
- ISO 8601 タイムスタンプフォーマットを使用する

## ステップ 13: 計画の明示的な承認を待機する
- ユーザーがストーリーアプローチを明示的に承認するまで進行しない
- 承認は明確で曖昧さのないものでなければならない
- ユーザーが変更を要求した場合は計画を更新して承認プロセスを繰り返す

## ステップ 14: 承認応答の記録
- `aidlc-docs/audit.md` にタイムスタンプ付きでユーザーの承認応答をログに記録する
- 正確なユーザー応答テキストを含める
- 承認ステータスを明確にマークする

## ステップ 20: 完了メッセージの提示
以下の構造で完了メッセージを提示する:

```markdown
# 📚 ユーザーストーリー完了
```

     2. **AI サマリー**（省略可）: 生成されたストーリーの構造化された箇条書きサマリーを提供する
        - フォーマット: 「ユーザーストーリー生成によって [説明] が作成されました:」

        - 数と組織化を含む作成されたユーザーストーリーをリストする

        - 事実のみで内容に集中する
     3. **フォーマット済みワークフローメッセージ**（必須）: 常にこの正確なフォーマットで終わる:

```markdown
> **📋 <u>**レビューをお願いします:**</u>**
> 以下のユーザーストーリーとペルソナを確認してください: `aidlc-docs/inception/user-stories/stories.md` および `aidlc-docs/inception/user-stories/personas.md`



> **🚀 <u>**次のステップ**</u>**
>
> **以下から選択してください:**
>
> 🔧 **変更を依頼** - レビュー内容に基づいてストーリーまたはペルソナの変更を依頼する
> ✅ **承認して続行** - ユーザーストーリーを承認して **ワークフロー計画** に進む

---
```

# 重要なルール

## 計画フェーズのルール
- **コンテキストに適した質問**: このコンテキストに関連する質問のみをする
- **必須の回答分析**: 進行する前に曖昧さについて常に回答を分析する
- **曖昧さのある状態で進行しない**: 生成前にすべての曖昧な回答を解消しなければならない
- **明示的な承認が必要**: 生成を開始する前にユーザーが計画を承認しなければならない

## 生成フェーズのルール
- **ハードコードされたロジックなし**: ストーリー生成計画に書かれていることのみを実行する
- **計画に厳密に従う**: ステップシーケンスから逸脱しない
- **チェックボックスを更新する**: 各ステップが完了したらすぐに [x] とマークする
- **承認された方法論を使用する**: 計画フェーズのストーリーアプローチに従う
- **完了を検証する**: 進行する前にすべてのストーリーアーティファクトが完全であることを確認する

## 完了基準
