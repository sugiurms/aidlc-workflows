# 要件分析（アダプティブ）

**プロダクトオーナーの役割を担う**

**アダプティブフェーズ**: 常に実行される。詳細レベルは問題の複雑さに適応する。

**アダプティブなデプスの説明については [depth-levels.md](../common/depth-levels.md) を参照**

## 前提条件
- ワークスペース検出が完了していること
- リバースエンジニアリングが完了していること（ブラウンフィールドの場合）

## 実行手順

### ステップ 1: リバースエンジニアリングコンテキストの読み込み（利用可能な場合）

**ブラウンフィールドプロジェクトの場合**:
- `aidlc-docs/inception/reverse-engineering/architecture.md` を読み込む
- `aidlc-docs/inception/reverse-engineering/component-inventory.md` を読み込む
- `aidlc-docs/inception/reverse-engineering/technology-stack.md` を読み込む
- リクエストを分析する際に既存システムを理解するためにこれらを使用する

### ステップ 2: ユーザーリクエストの分析（意図分析）

#### 2.1 リクエストの明確さ
- **明確**: 具体的、明確に定義されており、実行可能
- **あいまい**: 一般的、曖昧、要明確化
- **不完全**: 重要な情報が欠けている

#### 2.2 リクエストの種類
- **新機能**: 新しい機能の追加
- **バグ修正**: 既存の問題の修正
- **リファクタリング**: コード構造の改善
- **アップグレード**: 依存関係またはフレームワークの更新
- **マイグレーション**: 別のテクノロジーへの移行
- **機能強化**: 既存機能の改善
- **新プロジェクト**: ゼロから始める

#### 2.3 初期スコープの見積もり
- **単一ファイル**: 1 つのファイルへの変更
- **単一コンポーネント**: 1 つのコンポーネント/パッケージへの変更
- **複数コンポーネント**: 複数のコンポーネントにまたがる変更
- **システム全体**: システム全体に影響する変更
- **クロスシステム**: 複数のシステムに影響する変更

#### 2.4 初期複雑さの見積もり
- **ごく単純**: シンプルで直接的な変更
- **シンプル**: 明確な実装パス
- **中程度**: ある程度の複雑さ、複数の考慮事項
- **複雑**: 重大な複雑さ、多くの考慮事項

### ステップ 3: 要件のデプスの決定

**リクエスト分析に基づいてデプスを決定する:**

**最小デプス** — 以下の場合に使用:
- リクエストが明確でシンプルである
- 詳細な要件が不要
- 基本的な理解を文書化するだけでよい

**標準デプス** — 以下の場合に使用:
- リクエストに要明確化がある
- 機能要件と非機能要件が必要
- 通常の複雑さ

**包括的デプス** — 以下の場合に使用:
- 複数のステークホルダーがいる複雑なプロジェクト
- 高リスクまたはクリティカルなシステム
- トレーサビリティを含む詳細な要件が必要

### ステップ 4: 現在の要件の評価

ユーザーが提供したものを分析する:
   - 意図の記述または説明（すでに audit.md にログ記録済み）
   - 既存の要件ドキュメント（言及された場合はワークスペースで検索する）
   - 貼り付けられたコンテンツまたはファイルの参照
   - 非 Markdown ドキュメントを Markdown 形式に変換する

### ステップ 5: 徹底的な完全性分析

**重要**: 要件の完全性を評価するために包括的な分析を使用する。曖昧さや詳細の欠如がある場合は、デフォルトで質問する。

**必須**: これらすべての領域を評価し、不明確なものについては質問する:
- **機能要件**: コア機能、ユーザーインタラクション、システムの動作
- **非機能要件**: パフォーマンス、セキュリティ、スケーラビリティ、ユーザビリティ
- **ユーザーシナリオ**: ユースケース、ユーザージャーニー、エッジケース、エラーシナリオ
- **ビジネスコンテキスト**: 目標、制約、成功基準、ステークホルダーのニーズ
- **技術コンテキスト**: 統合ポイント、データ要件、システム境界
- **品質特性**: 信頼性、保守性、テスト可能性、アクセシビリティ

**疑問がある場合は質問する** — 不完全な要件は質の低い実装につながる。

### ステップ 5.1: 拡張機能オプトインプロンプト

**必須**: すべての読み込まれた `*.opt-in.md` ファイル（`extensions/` サブディレクトリからワークフロー開始時に読み込まれる）の `## Opt-In Prompt` セクションをスキャンする。宣言している各拡張機能について、ステップ 6 で作成する確認質問ファイルにその質問を含める。

回答を受け取った後:
1. 各拡張機能の有効化状態を `aidlc-docs/aidlc-state.md` の `## Extension Configuration` 下に記録する:

```markdown
## Extension Configuration
| Extension | Enabled | Decided At |
|---|---|---|
| [Extension Name] | [Yes/No] | Requirements Analysis |
```

2. **遅延ルール読み込み**: ユーザーがオプトインした各拡張機能について、今すぐ完全なルールファイルを読み込む。ルールファイルはのネーミング規則によって導出される: オプトインファイル名から `.opt-in.md` を取り除いて `.md` を追加する（例: `security-baseline.opt-in.md` → `security-baseline.md`）。ユーザーがオプトアウトした拡張機能については、完全なルールファイルを読み込まない。

### ステップ 6: 確認質問の生成（プロアクティブアプローチ）
   - 要件が例外的に明確で完全でない限り、**常に** `aidlc-docs/inception/requirements/requirement-verification-questions.md` を作成する
   - 欠けている、不明確な、または曖昧な領域についての質問をする
   - 機能要件、非機能要件、ユーザーシナリオ、ビジネスコンテキストに集中する
   - ユーザーがすべての `[Answer]:` タグを質問ドキュメントに直接記入するよう求める
   - 回答に複数選択のオプションを提示する場合:
     - オプションを A、B、C、D などとしてラベル付けする
     - オプションが相互に排他的で重複しないことを確認する
     - カスタム回答のオプションを**常に**含める: 「X) Other (please describe after [Answer]: tag below)」
   - ドキュメント内のユーザーの回答を待つ
   - **必須**: 曖昧さについてすべての回答を分析し、必要に応じてフォローアップ質問を作成する
   - **必須**: すべての曖昧さが解消されるか、ユーザーが明示的に進行を求めるまで質問を続ける

### ⛔ ゲート: ユーザーの回答を待機
requirement-verification-questions.md のすべての質問が回答されて検証されるまでステップ 7 に進まない。
質問ファイルをユーザーに提示して停止する。

### ステップ 7: 要件ドキュメントの生成
   - **前提条件**: ステップ 6 のゲートをパスしなければならない — すべての回答を受け取り分析済み
   - `aidlc-docs/inception/requirements/requirements.md` を作成する
   - 上部に意図分析サマリーを含める:
     - ユーザーのリクエスト
     - リクエストの種類
     - スコープの見積もり
     - 複雑さの見積もり
   - 機能要件と非機能要件の両方を含める
   - 確認質問へのユーザーの回答を組み込む
   - 主要な要件の簡潔なサマリーを提供する

### ステップ 8: 状態トラッキングの更新

`aidlc-docs/aidlc-state.md` を更新する:

```markdown
## Stage Progress
### 🔵 INCEPTION PHASE
- [x] Workspace Detection
- [x] Reverse Engineering (if applicable)
- [x] Requirements Analysis
```

### ステップ 9: ログ記録と進行
   - `aidlc-docs/audit.md` にタイムスタンプ付きで承認プロンプトをログに記録する
   - 以下の構造で完了メッセージを提示する:
     1. **完了告知**（必須）: 常にこれで始める:

```markdown
# 🔍 Requirements Analysis Complete
```

     2. **AI サマリー**（省略可）: 要件の構造化された箇条書きサマリーを提供する
        - フォーマット: 「要件分析によって [プロジェクトの種類/複雑さ] が特定されました:」
        - 主要な機能要件をリストする（箇条書き）
        - 主要な非機能要件をリストする（箇条書き）
        - 関連する場合はアーキテクチャの考慮事項や技術的な決定について言及する
        - ワークフローの指示を含めない（「レビューしてください」「お知らせください」「次のフェーズに進みます」「進める前に」）
        - 事実のみで内容に集中する
     3. **フォーマット済みワークフローメッセージ**（必須）: 常にこの正確なフォーマットで終わる:

```markdown
> **📋 <u>**REVIEW REQUIRED:**</u>**
> Please examine the requirements document at: `aidlc-docs/inception/requirements/requirements.md`



> **🚀 <u>**WHAT'S NEXT?**</u>**
>
> **You may:**
>
> 🔧 **Request Changes** -  Ask for modifications to the requirements if required based on your review
> [ユーザーストーリーがスキップされる場合はこのオプションを追加:]
> 📝 **Add User Stories** - Choose to Include **User Stories** stage (currently skipped based on project simplicity)
> ✅ **Approve & Continue** - Approve requirements and proceed to **[User Stories/Workflow Planning]**

---
```

**注記**: ユーザーストーリーステージがスキップされる場合にのみ「Add User Stories」オプションを含める。[User Stories/Workflow Planning] を実際の次のステージ名に置き換える。

   - 進行する前に明示的なユーザーの承認を待つ
   - タイムスタンプ付きで承認応答を記録する
