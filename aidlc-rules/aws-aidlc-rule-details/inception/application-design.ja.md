# アプリケーション設計 — 詳細な手順

## 目的
**高レベルのコンポーネント特定とサービス層設計**

アプリケーション設計が対象とする内容:
- 主要な機能コンポーネントとその責務の特定
- コンポーネントインターフェースの定義（詳細なビジネスロジックは含まない）
- オーケストレーションのためのサービス層の設計
- コンポーネントの依存関係とコミュニケーションパターンの確立

**注記**: 詳細なビジネスロジック設計は後の機能設計（ユニット単位、コンストラクションフェーズ）で行われます

## 前提条件
- ワークスペース検出が完了していること
- 要件分析を推奨（機能的なコンテキストを提供する）
- ユーザーストーリーを推奨（設計上の決定を導くユーザーストーリー）
- 実行計画がアプリケーション設計ステージを実行すべきことを示していること

## ステップバイステップの実行

### 1. コンテキストの分析
- `aidlc-docs/inception/requirements/requirements.md` と `aidlc-docs/inception/user-stories/stories.md` を読み込む
- 主要なビジネス機能と機能エリアを特定する
- 設計スコープと複雑さを決定する

### 2. アプリケーション設計計画の作成
- アプリケーション設計のためのチェックボックス [] を含む計画を生成する
- コンポーネント、責務、メソッド、ビジネスルール、サービスに集中する
- 各ステップとサブステップにチェックボックス [] を持たせる

### 3. 必須設計アーティファクトを計画に含める
- 以下の必須アーティファクトを設計計画に**常に**含める:
  - [ ] コンポーネント定義と高レベルの責務を含む components.md を生成する
  - [ ] メソッドシグネチャを含む component-methods.md を生成する（ビジネスルールの詳細は後で機能設計で行う）
  - [ ] サービス定義とオーケストレーションパターンを含む services.md を生成する
  - [ ] 依存関係とコミュニケーションパターンを含む component-dependency.md を生成する
  - [ ] 設計の完全性と一貫性を検証する

### 4. コンテキストに適した質問の生成
**ディレクティブ**: 要件とストーリーを分析して、この特定のアプリケーション設計に関連する質問を生成する。以下のカテゴリをガイダンスとして使用する。各カテゴリを評価し、適用性が不明な場合は省略するよりも質問する — 過信は悪い結果につながる（overconfidence-prevention.md 参照）。

- `[Answer]:` タグフォーマットを使用して質問を埋め込む
- 曖昧さ、欠けている情報、または明確化が必要な領域に集中する
- ユーザー入力が設計上の決定を改善するすべての箇所に質問を生成する
- **疑問がある場合は質問する** — 過信は設計の品質を下げる

**評価すべき質問カテゴリ**（すべてのカテゴリを考慮すること）:
- **コンポーネント特定** — コンポーネントの境界、組織化、グルーピング戦略について質問する
- **コンポーネントメソッド** — メソッドシグネチャ、入出力の期待値、インターフェースコントラクトについて質問する（詳細なビジネスルールは後で）
- **サービス層設計** — サービスのオーケストレーション、境界、コーディネーションパターンについて質問する
- **コンポーネント依存関係** — コミュニケーションパターン、依存関係管理、結合の懸念について質問する
- **設計パターン** — アーキテクチャスタイルの好み、パターンの選択、設計上の制約について質問する

### 5. アプリケーション設計計画の保存
- `aidlc-docs/inception/plans/application-design-plan.md` として保存する
- ユーザー入力のためのすべての `[Answer]:` タグを含める
- 計画がすべての設計側面をカバーすることを確認する

### 6. ユーザー入力の要求
- ユーザーが計画ドキュメント内の `[Answer]:` タグを直接記入するよう求める
- 設計上の決定の重要性を強調する
- `[Answer]:` タグの完了に関する明確な指示を提供する

### 7. 回答の収集
- ユーザーがドキュメント内の `[Answer]:` タグを使用してすべての質問に回答するまで待機する
- すべての `[Answer]:` タグが完了するまで進行しない
- `[Answer]:` タグが空白になっていないことをドキュメントで確認する

### 8. 回答の分析（必須）
進行する前に、すべてのユーザーの回答を慎重に確認しなければならない:
- **曖昧または不明確な応答**: 「混在」「どちらかというと」「わからない」「依存する」
- **未定義の基準または用語**: 明確な定義なしに概念を参照している
- **矛盾した回答**: 互いに矛盾する応答
- **設計の詳細が欠けている**: 具体的なガイダンスがない回答
- **オプションを組み合わせている回答**: 明確な決定ルールなしに異なるアプローチを統合した応答

### 9. 必須フォローアップ質問
ステップ 8 の分析で曖昧な回答が明らかになった場合、以下を行わなければならない:
- `[Answer]:` タグを使用して計画ドキュメントに具体的なフォローアップ質問を追加する
- すべての曖昧さが解消されるまで承認プロセスに進まない
- 必要なフォローアップの例:
  - 「A と B の混在とおっしゃいましたが、A と B をいつ使うかを決める具体的な基準は何ですか?」
  - 「A と B の中間とおっしゃいましたが、正確な中間アプローチを定義していただけますか?」
  - 「不明とおっしゃいましたが、何の追加情報があれば決定できますか?」
  - 「複雑さによって異なるとおっしゃいましたが、複雑さのレベルをどう定義しますか?」

### 10. アプリケーション設計アーティファクトの生成
- 承認された計画を実行して設計アーティファクトを生成する
- `aidlc-docs/inception/application-design/components.md` を作成する（含む内容）:
  - コンポーネント名と目的
  - コンポーネントの責務
  - コンポーネントインターフェース
- `aidlc-docs/inception/application-design/component-methods.md` を作成する（含む内容）:
  - 各コンポーネントのメソッドシグネチャ
  - 各メソッドの高レベルな目的
  - 入出力の型
  - 注記: 詳細なビジネスルールは機能設計（ユニット単位、コンストラクションフェーズ）で定義される
- `aidlc-docs/inception/application-design/services.md` を作成する（含む内容）:
  - サービス定義
  - サービスの責務
  - サービスのインタラクションとオーケストレーション
- `aidlc-docs/inception/application-design/component-dependency.md` を作成する（含む内容）:
  - 関係を示す依存関係マトリクス
  - コンポーネント間のコミュニケーションパターン
  - データフロー図
- 上記で作成された複数の設計ドキュメントを 1 つのドキュメントに統合した `aidlc-docs/inception/application-design/application-design.md` を作成する

### 11. 承認のログ記録
- `aidlc-docs/audit.md` にタイムスタンプ付きで承認プロンプトをログに記録する
- 完全な承認プロンプトテキストを含める
- ISO 8601 タイムスタンプフォーマットを使用する

### 12. 完了メッセージの提示

```markdown
# 🏗️ アプリケーション設計完了

[作成されたアプリケーション設計アーティファクトの AI 生成サマリー（箇条書き）]

> **📋 <u>**レビューをお願いします:**</u>**
> 以下のアプリケーション設計成果物を確認してください: `aidlc-docs/inception/application-design/`

> **🚀 <u>**次のステップ**</u>**
>
> **以下から選択してください:**
>
> 🔧 **変更を依頼** - 必要に応じてアプリケーション設計の変更を依頼する
> [ユニット生成がスキップされる場合:]
> 📝 **ユニット生成を追加** - **ユニット生成** ステージを含める（現在スキップされています）
> ✅ **承認して続行** - 設計を承認して **[ユニット生成/CONSTRUCTION フェーズ]** に進む
```

### 13. 明示的な承認の待機
- ユーザーがアプリケーション設計を明示的に承認するまで進行しない
- 承認は明確で曖昧さのないものでなければならない
- ユーザーが変更を要求した場合は設計を更新して承認プロセスを繰り返す

### 14. 承認応答の記録
- `aidlc-docs/audit.md` にタイムスタンプ付きでユーザーの承認応答をログに記録する
- 正確なユーザー応答テキストを含める
- 承認ステータスを明確にマークする

### 15. 進捗の更新
- `aidlc-docs/aidlc-state.md` でアプリケーション設計ステージを完了としてマークする
- 「現在のステータス」セクションを更新する
- 次のステージへの移行の準備をする
