# ユニット生成 — 詳細な手順

## 概要
このステージでは、2 つの統合パートを通じてシステムを管理可能な作業単位に分解します:
- **パート 1 — 計画**: 質問を含む分解計画を作成し、回答を収集し、曖昧さを分析して承認を得る
- **パート 2 — 生成**: 承認された計画を実行してユニットアーティファクトを生成する

**定義**: 作業単位とは、開発目的で論理的にグループ化されたストーリーです。マイクロサービスの場合、各ユニットは独立してデプロイ可能なサービスになります。モノリスの場合、単一ユニットは論理モジュールを持つアプリケーション全体を表します。

**用語**: 独立してデプロイ可能なコンポーネントには「Service（サービス）」、サービス内の論理グループには「Module（モジュール）」、計画コンテキストには「Unit of Work（作業単位）」を使用する。

## 前提条件
- ワークスペース検出が完了していること
- 要件分析を推奨（機能スコープを提供する）
- ユーザーストーリーを推奨（ストーリーはユニットにマッピングされる）
- アプリケーション設計ステージが**必須**（コンポーネント、メソッド、サービスを決定する）
- 実行計画が設計ステージを実行すべきことを示していること

# パート 1: 計画

## ステップ 1: 作業単位計画の作成
- システムを作業単位に分解するためのチェックボックス [] を含む計画を生成する
- システムを管理可能な開発単位に分解することに集中する
- 各ステップとサブステップにチェックボックス [] を持たせる

## ステップ 2: 必須ユニットアーティファクトを計画に含める
以下の必須アーティファクトをユニット計画に**常に**含める:
- [ ] ユニット定義と責務を含む `aidlc-docs/inception/application-design/unit-of-work.md` を生成する
- [ ] 依存関係マトリクスを含む `aidlc-docs/inception/application-design/unit-of-work-dependency.md` を生成する
- [ ] ストーリーをユニットにマッピングする `aidlc-docs/inception/application-design/unit-of-work-story-map.md` を生成する
- [ ] **グリーンフィールドのみ**: コード組織化戦略を `unit-of-work.md` に文書化する（構造パターンについては code-generation.md を参照）
- [ ] ユニットの境界と依存関係を検証する
- [ ] すべてのストーリーがユニットに割り当てられていることを確認する

## ステップ 3: コンテキストに適した質問の生成
**ディレクティブ**: 要件、ストーリー、アプリケーション設計を徹底的に分析して、確認によってユニット分解の品質が向上するすべての領域を特定する。分解に関わる懸念事項の包括的なカバレッジを確保するために積極的に質問する。

**重要**: ユニットの境界や分解の品質に影響する可能性のある曖昧さや詳細の欠如がある場合は、デフォルトで質問する。システムを分解する方法について誤った想定をするよりも質問が多すぎる方が良い。

**必須**: 以下のすべてのカテゴリを評価し、各カテゴリに対して的を絞った質問をする。各カテゴリの適用可能性は要件、ストーリー、アプリケーション設計から判断する — 明確な理由なくカテゴリをスキップしない:

- `[Answer]:` タグフォーマットを使用して質問を埋め込む
- 曖昧さ、欠けている情報、または明確化が必要な領域に集中する
- ユーザー入力が分解の決定を改善するすべての箇所に質問を生成する
- **疑問がある場合は質問する** — 過信は不適切なユニット境界につながる

**評価すべき質問カテゴリ**（すべてのカテゴリを考慮すること）:
- **ストーリーのグルーピング** — グルーピング戦略、ストーリーの親和性、論理的なクラスタリングアプローチについて質問する
- **依存関係** — 統合アプローチ、共有リソース、ユニット間のコミュニケーションパターンについて質問する
- **チームの整合性** — チーム構造、所有権の境界、コラボレーションモデルについて質問する
- **技術的な考慮事項** — ユニット間で異なる可能性のあるスケーラビリティ/デプロイ要件について質問する
- **ビジネスドメイン** — ドメインの境界、境界付けられたコンテキスト、ビジネス能力の整合性について質問する
- **コード組織化（グリーンフィールドのマルチユニットのみ）** — デプロイモデルとディレクトリ構造の好みについて質問する

## ステップ 4: 作業単位計画の保存
- `aidlc-docs/inception/plans/unit-of-work-plan.md` として保存する
- ユーザー入力のためのすべての `[Answer]:` タグを含める
- 計画がシステム分解のすべての側面をカバーすることを確認する

## ステップ 5: ユーザー入力の要求
- ユーザーが計画ドキュメント内の `[Answer]:` タグを直接記入するよう求める
- 分解の決定の重要性を強調する
- `[Answer]:` タグの完了に関する明確な指示を提供する

## ステップ 6: 回答の収集
- ユーザーがドキュメント内の `[Answer]:` タグを使用してすべての質問に回答するまで待機する
- すべての `[Answer]:` タグが完了するまで進行しない
- `[Answer]:` タグが空白になっていないことをドキュメントで確認する

## ステップ 7: 回答の分析（必須）
進行する前に、すべてのユーザーの回答を慎重に確認しなければならない:
- **曖昧または不明確な応答**: 「混在」「どちらかというと」「わからない」「依存する」
- **未定義の基準または用語**: 明確な定義なしに概念を参照している
- **矛盾した回答**: 互いに矛盾する応答
- **生成の詳細が欠けている**: 具体的なガイダンスがない回答
- **オプションを組み合わせている回答**: 明確な決定ルールなしに異なるアプローチを統合した応答

## ステップ 8: 必須フォローアップ質問
ステップ 7 の分析で曖昧な回答が明らかになった場合、以下を行わなければならない:
- `[Answer]:` タグを使用して計画ドキュメントに具体的なフォローアップ質問を追加する
- すべての曖昧さが解消されるまで承認プロセスに進まない
- 必要なフォローアップの例:
  - 「A と B の混在とおっしゃいましたが、A と B をいつ使うかを決める具体的な基準は何ですか?」
  - 「A と B の中間とおっしゃいましたが、正確な中間アプローチを定義していただけますか?」
  - 「不明とおっしゃいましたが、何の追加情報があれば決定できますか?」
  - 「複雑さによって異なるとおっしゃいましたが、複雑さのレベルをどう定義しますか?」

## ステップ 9: 承認の要求
- 次のように尋ねる: 「**作業単位計画が完成しました。aidlc-docs/inception/plans/unit-of-work-plan.md の計画を確認してください。生成に進む準備ができましたか?**」
- ユーザーが確認するまで**進行しない**

## ステップ 10: 承認のログ記録
- タイムスタンプ付きで audit.md にプロンプトと応答をログに記録する
- ISO 8601 タイムスタンプフォーマットを使用する
- 完全な承認プロンプトテキストを含める

## ステップ 11: 進捗の更新
- aidlc-state.md でユニット計画を完了としてマークする
- 「現在のステータス」セクションを更新する
- ユニット生成への移行の準備をする

# パート 2: 生成

## ステップ 12: 作業単位計画の読み込み
- [ ] `aidlc-docs/inception/plans/unit-of-work-plan.md` から完全な計画を読み込む
- [ ] 次の未完了のステップ（最初の [ ] チェックボックス）を特定する
- [ ] そのステップのコンテキストと要件を読み込む

## ステップ 13: 現在のステップの実行
- [ ] 現在のステップが記述する内容を正確に実行する
- [ ] 計画で指定されたとおりにユニットアーティファクトを生成する
- [ ] 計画フェーズで承認された分解アプローチに従う
- [ ] 計画で指定された基準と境界を使用する

## ステップ 14: 進捗の更新
- [ ] 完了したステップを作業単位計画で [x] としてマークする
- [ ] `aidlc-docs/aidlc-state.md` の現在のステータスを更新する
- [ ] 生成されたすべてのアーティファクトを保存する

## ステップ 15: 継続または完了
- [ ] さらにステップが残っている場合はステップ 12 に戻る
- [ ] すべてのステップが完了したら、ユニットが設計ステージの準備ができていることを確認する
- [ ] ユニット生成ステージを完了としてマークする

## ステップ 16: 完了メッセージの提示

```markdown

[作成されたユニットと分解の AI 生成サマリー（箇条書き）]

> 以下のユニット生成成果物を確認してください: `aidlc-docs/inception/application-design/`

> **🚀 <u>**次のステップ**</u>**
>
> **以下から選択してください:**
>
> 🔧 **変更を依頼** - 必要に応じてユニット生成の変更を依頼する
> ✅ **承認して続行** - ユニットを承認して **CONSTRUCTION フェーズ** に進む
```

## ステップ 17: 明示的な承認の待機
- ユーザーがユニット生成を明示的に承認するまで進行しない
- 承認は明確で曖昧さのないものでなければならない
- ユーザーが変更を要求した場合はユニットを更新して承認プロセスを繰り返す

## ステップ 18: 承認応答の記録
- `aidlc-docs/audit.md` にタイムスタンプ付きでユーザーの承認応答をログに記録する
- 正確なユーザー応答テキストを含める
- 承認ステータスを明確にマークする

## ステップ 19: 進捗の更新
- `aidlc-docs/aidlc-state.md` でユニット生成ステージを完了としてマークする
- 「現在のステータス」セクションを更新する
- コンストラクションフェーズへの移行の準備をする

## 重要なルール

### 計画フェーズのルール
- コンテキストに関連する質問のみを生成する
- すべての質問に `[Answer]:` タグフォーマットを使用する
- 進行する前に曖昧さについてすべての回答を分析する
- フォローアップ質問ですべての曖昧さを解消する
- 生成前に明示的なユーザーの承認を得る

### 生成フェーズのルール
- **ハードコードされたロジックなし**: 作業単位計画に書かれていることのみを実行する
- **計画に厳密に従う**: ステップシーケンスから逸脱しない
- **チェックボックスを更新する**: 各ステップが完了したらすぐに [x] とマークする
- **承認されたアプローチを使用する**: 計画フェーズの分解方法論に従う
- **完了を検証する**: 進行する前にすべてのユニットアーティファクトが完全であることを確認する

## 完了基準
- すべての計画の質問が回答され、曖昧さが解消されている
- 計画に対してユーザーの承認が得られている
- 作業単位計画のすべてのステップが [x] とマークされている
- 計画に従ってすべてのユニットアーティファクトが生成されている:
  - ユニット定義を含む `unit-of-work.md`
  - 依存関係マトリクスを含む `unit-of-work-dependency.md`
  - ストーリーマッピングを含む `unit-of-work-story-map.md`
- ユニットがユニット単位の設計ステージの準備ができており検証されている
